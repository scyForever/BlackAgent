下面给一个合并后的完整方案：它同时覆盖你前面问的“项目代码怎么重构”、以及“效果 / 成本 / 时延三角平衡、LLM + NLP 协同、Agent 编排怎么优化”。

核心结论先放前面：

```text
不要推倒重写。
不要把所有能力都 LLM 化。
不要把 Agent 做成自由探索型。

推荐方向是：
受控工作流 DAG + NLP/规则批处理 + LLM 受控增强 + Agent 预算编排 + 人工复核闭环。
```

你的课题目标本身是构建“情报采集 → 智能清洗 → 意图分类 → 实体抽取”的端到端流水线，产出风险线索、风险样本和结构化实体库；同时核心挑战明确包括多源异构、噪声重复、黑话隐蔽，以及效果 / 成本 / 时延三角平衡。这个完整方案就围绕这条主线展开。

# 1. 总体目标

你的系统最终应该从现在的：

```text
CLI / LocalRuntime
  ↓
InvestigationOrchestrator 大编排
  ↓
collector / cleaner / classifier / extractor / storage / LLM 混合调用
```

重构为：

```text
Interface 层：CLI / API / Web Console
  ↓
Application 层：InvestigationService / TaskService / ReviewService
  ↓
Agent 编排层：Orchestrator / Planner / SourceSelector / ModelRouter
  ↓
Pipeline 层：Clean → Dedup → Classify → Extract → Correlate → Score
  ↓
Domain 层：RawIntelligence / CleanedText / RiskClassification / Entity / RiskClue
  ↓
Infra 层：Storage / LLMGateway / Collector / Config / Telemetry
  ↓
Safety 层：PolicyGuard / PromptGuard / OutputValidator / Enforcement
```

也就是把系统分成三类能力：

```text
确定性能力：
  清洗、去重、规则分类、实体正则抽取、schema 校验、风险评分。

智能增强能力：
  黑话语义理解、隐晦表达判断、冲突分类裁决、复杂线索摘要。

编排治理能力：
  预算控制、模型路由、source 选择、超时降级、人工复核、审计追踪。
```

这三类能力必须解耦。否则后面你越加功能，`InvestigationOrchestrator`、`LocalRuntime`、`SQLBackend` 会继续膨胀。

推荐目标架构如下：

```text
┌─────────────────────────────────────────────┐
│ Interface Layer                              │
│ CLI / API / Web Console                      │
└──────────────────────┬──────────────────────┘
                       ↓
┌─────────────────────────────────────────────┐
│ Application Layer                            │
│ InvestigationService / TaskService / Review  │
└──────────────────────┬──────────────────────┘
                       ↓
┌─────────────────────────────────────────────┐
│ Agent Orchestration Layer                    │
│ Orchestrator / Planner / SourceSelector      │
│ ModelRouter / BudgetController               │
└──────────────────────┬──────────────────────┘
                       ↓
┌─────────────────────────────────────────────┐
│ Intelligence Pipeline                        │
│ Clean → Dedup → Triage → Classify → Extract  │
│ → Normalize → Correlate → Score              │
└──────────────────────┬──────────────────────┘
                       ↓
┌─────────────────────────────────────────────┐
│ Knowledge & Storage Layer                    │
│ Raw Store / Entity DB / Clue DB / Vector DB  │
│ Graph DB / Review DB / Audit DB              │
└──────────────────────┬──────────────────────┘
                       ↓
┌─────────────────────────────────────────────┐
│ Safety & Governance Layer                    │
│ PolicyGuard / PromptGuard / OutputValidator  │
│ PII Masking / Audit / Human Review           │
└─────────────────────────────────────────────┘
```

核心设计原则：

```text
1. Agent 只负责编排，不直接承担所有业务逻辑。
2. Pipeline 只负责数据处理，不负责 source 决策。
3. LLM 只做增强，不做默认主路径。
4. Storage 只负责持久化，不承载业务决策。
5. Safety 是横切层，所有入口和动作都要经过边界控制。
```

# 5. 标准执行流程

最终一次 investigation 应该按这个流程走：

```text
1. 接收用户请求
2. PolicyGuard 检查请求是否合法
3. IntentParser 解析目标
4. RoutingProfile 决定 fast / balanced / high_recall
5. 优先查已有 clue pool
6. 如果已有高质量线索足够：
      少量 refine 或直接返回
   否则：
      选择授权 source
      执行采集
7. Raw records 进入 IntelligencePipeline
8. CleanStage 清洗
9. DedupStage 去重
10. TriageStage 风险初筛
11. RuleClassifier 规则分类
12. EntityExtractor 实体抽取
13. ModelRouter 判断哪些样本需要 LLM
14. LLM 只处理高价值、低确定性样本
15. CorrelateStage 做实体关联和线索聚合
16. ScoreStage 风险评分
17. ClueRanker 排序 Top-K 线索
18. LLMClueRefiner 批量精炼必要线索
19. ReviewRouter 决定人工复核 / 归档 / 告警
20. 写入 storage、telemetry、audit
```

这条流程的关键是：

```text
先查已有线索池。
先走规则和 NLP。
先做去重和聚合。
最后才让 LLM 介入。
```

# 7. LLM + NLP 协同方案

你的主路径应该是：

```text
规则 / NLP / 检索 / 聚类
  ↓
只把不确定但高价值的部分交给 LLM
```

具体分工如下。

## 7.1 规则和 NLP 负责什么

```text
1. 文本清洗
2. 去重
3. URL / 域名 / 联系方式识别
4. 账号 / 群号 / 工具名抽取
5. 明确关键词命中
6. 简单风险分类
7. 噪声过滤
8. 模板聚类
9. 低成本召回
10. 规则可解释证据输出
```

## 7.2 LLM 负责什么

```text
1. 隐晦表达识别
2. 黑话变种解释
3. 多标签冲突裁决
4. 复杂语义分类
5. 复杂实体归一化
6. 多证据线索摘要
7. 线索报告生成
8. 人工复核辅助说明
```

## 7.3 LLM 不应该负责什么

```text
1. 不负责默认清洗所有文本
2. 不负责逐条分析所有样本
3. 不负责直接访问 source
4. 不负责决定是否绕过边界
5. 不负责执行处置动作
6. 不负责补全原文没有的信息
7. 不负责生成黑灰产操作方案
```

# 8. 新增核心模块：ModelRouter

这是整个优化方案的关键。

`ModelRouter` 的职责不是调用模型，而是决定：

```text
这条样本是否需要 LLM？
需要哪类 LLM 能力？
最多花多少 token？
是否需要人工复核？
超时后如何降级？
```

建议新增：

```text
src/agent/model_router.py
```

核心动作分成四类：

```text
skip
  低质量、低风险、重复样本，跳过或归档。

deterministic_only
  规则 / 实体抽取结果足够确定，不调用 LLM。

llm_classify_extract
  高风险但语义不确定，调用 LLM 做分类 / 抽取增强。

llm_refine_only
  已经形成线索，只让 LLM 做摘要、证据组织、复核辅助。
```

示例逻辑：

```python
from dataclasses import dataclass
from typing import Literal

RouteAction = Literal[
    "skip",
    "deterministic_only",
    "llm_classify_extract",
    "llm_refine_only",
]


@dataclass(frozen=True)
class ModelRouteDecision:
    action: RouteAction
    reason: str
    priority: int
    max_tokens: int
    deadline_ms: int
    requires_review: bool


class ModelRouter:
    def __init__(self, profile: str = "balanced") -> None:
        self.profile = profile

    def decide_record(
        self,
        *,
        rule_confidence: float,
        risk_score: float,
        entity_count: int,
        has_contact: bool,
        has_url: bool,
        has_tool: bool,
        has_conflict: bool,
        is_duplicate: bool,
        quality_score: float,
    ) -> ModelRouteDecision:
        if quality_score < 0.25 and not (has_contact or has_url or has_tool):
            return ModelRouteDecision(
                action="skip",
                reason="low_quality_low_signal",
                priority=0,
                max_tokens=0,
                deadline_ms=0,
                requires_review=False,
            )

        if is_duplicate and rule_confidence >= 0.80:
            return ModelRouteDecision(
                action="deterministic_only",
                reason="duplicate_high_rule_confidence",
                priority=1,
                max_tokens=0,
                deadline_ms=0,
                requires_review=False,
            )

        if rule_confidence >= 0.85 and entity_count >= 2 and not has_conflict:
            return ModelRouteDecision(
                action="deterministic_only",
                reason="high_confidence_rule_and_entities",
                priority=2,
                max_tokens=0,
                deadline_ms=0,
                requires_review=False,
            )

        if has_conflict or ((has_contact or has_url or has_tool) and rule_confidence >= 0.45):
            return ModelRouteDecision(
                action="llm_classify_extract",
                reason="ambiguous_high_value_signal",
                priority=5,
                max_tokens=600 if self.profile == "fast" else 900,
                deadline_ms=2500 if self.profile == "fast" else 6000,
                requires_review=True,
            )

        return ModelRouteDecision(
            action="deterministic_only",
            reason="low_ambiguity_default",
            priority=1,
            max_tokens=0,
            deadline_ms=0,
            requires_review=rule_confidence < 0.70,
        )
```

这个模块接入后，系统会从：

```text
各模块自己决定是否调用 LLM
```

变成：

```text
所有 LLM 调用都必须经过统一路由决策
```

这是成本和时延可控的基础。

三个模式的定位：

```text

  面向交互式查询。
  优先查已有 clue pool。

  默认模式。
  允许少量采集、少量 query rewrite、少量 LLM refine。

high_recall：
  面向离线批处理、专项排查。
  放宽召回阈值，扩大 source 和样本量。
  但所有低置信高风险结果必须进入人工复核。
```

# 10. IntelligencePipeline 设计

建议新增：

```text
src/pipeline/intelligence_pipeline.py
```

标准 pipeline：

```python
class IntelligencePipeline:
    def __init__(
        self,
        clean_stage,
        dedup_stage,
        triage_stage,
        classify_stage,
        extract_stage,
        correlate_stage,
        score_stage,
        model_router,
    ) -> None:
        self.clean_stage = clean_stage
        self.dedup_stage = dedup_stage
        self.triage_stage = triage_stage
        self.classify_stage = classify_stage
        self.extract_stage = extract_stage
        self.correlate_stage = correlate_stage
        self.score_stage = score_stage
        self.model_router = model_router

    def run(self, raw_items, context):
        cleaned = self.clean_stage.run_batch(raw_items, context=context)
        deduped = self.dedup_stage.run_batch(cleaned, context=context)
        triaged = self.triage_stage.run_batch(deduped, context=context)
        classified = self.classify_stage.run_batch(triaged, context=context)
        extracted = self.extract_stage.run_batch(classified, context=context)

        routed = [
            self.model_router.decide_record(**item.routing_features())
            for item in extracted
        ]

        correlated = self.correlate_stage.run_batch(extracted, routed, context=context)
        scored = self.score_stage.run_batch(correlated, context=context)

        return PipelineResult(
            cleaned=cleaned,
            classified=classified,
            entities=extracted,
            routed=routed,
            clues=scored,
        )
```

Pipeline 的原则：

```text
每个 stage 只做一件事。
每个 stage 输入输出都是明确 schema。
每个 stage 不直接访问 CLI。
每个 stage 不直接做最终报告。
每个 stage 不绕过 safety。
```

```python
from pydantic import BaseModel, Field
from typing import Any, Literal


class RawIntelligence(BaseModel):
    record_id: str
    source_id: str
    source_type: str
    source_ref: str | None = None
    author_hash: str | None = None
    publish_time: str | None = None
    crawl_time: str | None = None
    raw_text: str
    raw_hash: str
    metadata: dict[str, Any] = Field(default_factory=dict)


class CleanedText(BaseModel):
    message_id: str
    record_id: str
    raw_text: str
    clean_text: str
    normalized_text: str
    language: str = "zh"
    quality_score: float
    duplicate_of: str | None = None
    risk_signal_profile: dict[str, Any] = Field(default_factory=dict)


class RiskClassification(BaseModel):
    message_id: str
    risk_type: str
    risk_subtype: str | None = None
    confidence: float
    severity: Literal["low", "medium", "high", "critical"]
    evidence_spans: list[str] = Field(default_factory=list)
    classifier: str
    model_version: str | None = None
    label_version: str | None = None
    review_required: bool = False


class ExtractedEntity(BaseModel):
    entity_id: str
    message_id: str
    entity_type: str
    raw_value: str
    masked_value: str | None = None
    normalized_value: str | None = None
    confidence: float
    sensitivity_level: Literal["public", "internal", "sensitive", "high_sensitive"]
    evidence_span: str | None = None
    extractor: str


class RiskClue(BaseModel):
    clue_id: str
    clue_type: str
    risk_score: float
    severity: Literal["low", "medium", "high", "critical"]
    confidence: float
    related_messages: list[str]
    related_entities: list[str]
    evidence_summary: str
    review_status: Literal["pending", "approved", "rejected", "archived"] = "pending"
    model_traces: list[dict[str, Any]] = Field(default_factory=list)
```

这些 schema 应该成为所有模块之间的通用契约，逐步减少 `dict[str, Any]` 乱传。

# 13. BudgetController 设计

你已有 `BudgetManager`，但它现在更像探索轮次预算。建议升级为全局运行预算。

```python
from dataclasses import dataclass


@dataclass
class RuntimeBudget:
    max_elapsed_seconds: int
    max_sources: int
    max_raw_records: int
    max_candidate_clues: int
    max_llm_calls: int
    max_llm_tokens: int
    max_llm_classify_records: int
    max_llm_extract_records: int
    max_llm_refine_clues: int
    max_query_rewrite_sources: int


class BudgetController:
    def __init__(self, budget: RuntimeBudget) -> None:
        self.budget = budget
        self.llm_calls = 0
        self.estimated_tokens = 0
        self.refined_clues = 0
        self.classified_by_llm = 0
        self.extracted_by_llm = 0

    def allow_llm_call(self, *, stage: str, estimated_tokens: int) -> bool:
        if self.llm_calls + 1 > self.budget.max_llm_calls:
            return False
        if self.estimated_tokens + estimated_tokens > self.budget.max_llm_tokens:
            return False
        return True

    def consume_llm(self, *, stage: str, estimated_tokens: int) -> None:
        self.llm_calls += 1
        self.estimated_tokens += estimated_tokens
```

BudgetController 要作为硬约束，而不是日志建议。

# 14. 线索 refine 优化

你当前最需要优化的 LLM 成本点之一，是线索逐条 refine。

应从：

```text
按原顺序逐条 refine
```

改为：

```text
先排序
再筛选
再 batch refine
```

## 14.1 线索 refine 排序

新增：

```text
src/agent/clue_ranker.py
```

排序分数：

```text
refine_priority =
  0.35 * quality_score
+ 0.25 * confidence
+ 0.20 * evidence_count_score
+ 0.10 * cross_source_score
+ 0.10 * entity_sensitivity_score
- 0.20 * already_high_quality_penalty
```

优先 refine：

```text
1. 风险高但摘要弱
2. 证据多但结构混乱
3. 分类冲突
4. 跨 source 传播
5. 高敏实体较多
6. 需要人工复核的重点线索
```

不 refine：

```text
1. 已经高质量且证据完整
2. 低风险低置信线索
3. 没有关键实体的线索
4. 重复模板线索
```

## 14.2 Batch refine

从：

```text
1 clue = 1 LLM call
```

改成：

```text
5 clues = 1 LLM call
```

LLM 输出：

```json
{
  "items": [
    {
      "clue_id": "clue_001",
      "refined_summary": "疑似账号交易引流线索，同一联系方式跨多个来源出现。",
      "confidence_delta": 0.05,
      "review_required": true,
      "refinement_reasons": [
        "跨来源重复",
        "存在联系方式",
        "命中账号交易标签"
      ]
    }
  ]
}
```

这样会同时降低成本和 p95 时延。

# 15. Agent 编排重构

`InvestigationOrchestrator` 不应该继续承担所有职责。建议逐步拆成：

```text
InvestigationOrchestrator
  只负责串联流程

IntentService
  解析用户目标和风险类型

PlanningService
  生成调查计划

SourceSelectionService
  选择 source

CollectionRunner
  调 collector 获取 raw records

PipelineRunner
  调 IntelligencePipeline

ClueBuilder
  生成候选线索

ClueRanker
  对线索排序

ClueRefinementService
  批量 refine

ReviewRouter
  决定复核 / 归档 / 告警

TelemetryRecorder
  记录调用链路、耗时、成本、效果
```

重构后的 orchestrator 应该像这样：

```python
class InvestigationOrchestrator:
    def run(self, request):
        policy_result = self.policy_guard.check_request(request)
        if not policy_result.allowed:
            return InvestigationResult.blocked(policy_result.reason)

        intent = self.intent_service.parse(request)
        profile = self.profile_resolver.resolve(request, intent)
        budget = self.budget_factory.create(profile)

        existing_clues = self.clue_retriever.retrieve(intent, budget)

        if self.early_stop.should_return_from_pool(existing_clues, profile):
            refined = self.clue_refinement_service.refine_topk(
                existing_clues,
                intent=intent,
                budget=budget,
            )
            return self.result_builder.build(refined)

        plan = self.planning_service.create(intent, profile, budget)
        sources = self.source_selector.select(plan, budget)

        raw_records = self.collection_runner.collect(sources, budget)
        pipeline_result = self.pipeline_runner.run(raw_records, intent, profile, budget)

        clues = self.clue_builder.build(pipeline_result)
        ranked = self.clue_ranker.rank(clues)
        refined = self.clue_refinement_service.refine_topk(ranked, intent=intent, budget=budget)

        routed = self.review_router.route(refined)
        self.telemetry.record(request, routed, budget)

        return self.result_builder.build(routed)
```

这个结构的好处是：每个模块都能单独测试，Orchestrator 不再是“上帝类”。

# 22. 评估体系

建议建立四套 golden set。

```text
gold_classification_set
  人工标注风险类型和子类型。

gold_entity_set
  人工标注账号、链接、联系方式、工具、黑话等实体。

hard_negative_set
  新闻报道、反诈提醒、研究文章、正常交易等易误报样本。

slang_variant_set
  黑话、谐音、拆字、emoji、拼音缩写样本。
```

每次修改规则、prompt、阈值、模型路由，都要跑评估：

```text
分类 precision / recall / F1
实体抽取 precision / recall / F1
高危召回率
误报率
有效线索率
人工复核通过率
LLM calls / 1000 records
tokens / valid clue
p95 latency
```

目标不是单点最优，而是找到 Pareto 最优：

```text
在召回率基本不降的情况下，减少 LLM 调用。
在成本可控的情况下，提高有效线索率。
在 p95 可接受的情况下，扩大覆盖源。
```

# 23. 最后


最终你要达成的系统状态是：

```text
NLP / 规则处理 80% 的确定性工作；
LLM 只处理 20% 的高价值不确定问题；
Agent 不自由探索，而是在预算、边界、质量门槛内编排；
所有结果可追溯、可复核、可评估；
所有 LLM 调用可计量、可缓存、可降级；
所有敏感行为受 PolicyGuard 和 Enforcement 控制。
```

这就是当前项目最合理的完整重构和优化方案。
