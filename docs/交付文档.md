# BlackAgent 交付文档

> 本文档按 `黑灰产情报分析Agent.docx` 中的课题目标、分阶段目标和核心挑战逐项说明：当前仓库如何实现、交付了什么产物、实际产出结果是什么。所有数字均来自当前仓库已有产物文件或已验证命令，不把规划项写成已上线能力。

## 1. 课题原始要求摘要

Word 文档中的题目和简介可以概括为：

- 题目：黑灰产情报分析 Agent。
- 简介：通过获取 IM、群组、公众号、论坛等多源渠道汇聚的海量数据，完成数据清洗、高危意图分类，抽取黑话、链接、账号、工具等关键实体，产出风险线索和风险样本。

Word 中的总体目标是：

```text
构建一条“情报采集 -> 智能清洗 -> 意图分类 -> 实体抽取”的端到端自动化分析流水线，帮助业务从“人肉研判”走向“智能对抗”。
```

Word 中的总体交付口径是：

```text
把散落的黑灰产情报，自动变成结构化的作弊剧本和对抗方案。
```

当前仓库的实际落地形态按 5 阶段对外表达：

```text
输入任务
  -> 安全与任务路由
  -> 历史资产检索
  -> 情报处理流水线
  -> 线索生成与报告
```

展开后对应到真实产物：

```text
用户 query / 上传样本 / 指定 source / 时间范围
  -> PolicyGuard / SourcePolicyGuard / routing profile / 预算控制
  -> clue pool / entity graph / semantic cache 检索
  -> Need Fresh Data?
       ├─ No  -> candidate/actionable clue promotion -> report
       └─ Yes -> 公开/授权 source 采集或用户输入 records
                  -> Clean / Dedup / Classify / Extract / Normalize / EntityGraph
                  -> candidate_clues / actionable_clues / high_quality_clues
                  -> 可复核 JSON、统计报表和 CLI 调查结果
```

### 3.2.2 当前实现方式

主要代码：

| 文件 | 作用 |
| --- | --- |
| `src/cleaner/text_filter.py` | 文本归一、噪声判断、熵值、去重指纹、风险信号、质量分 |
| `src/cleaner/pipeline.py` | 批量清洗 pipeline |

| `config/theme_synonyms.yaml` | 主题同义词和搜索词扩展 |
| `scripts/run_cleaning_phase.py` | 清洗阶段脚本 |
| `storage/sql_backend.py` | `cleaned_texts` 持久化 |

处理链路：

```text
raw_records
  -> 多模态文本物化
  -> normalize_text / normalize_intel_text
  -> blank / garbled / entropy 判断
  -> canonicalize_for_dedup
  -> stable_dedup_group_id
  -> duplicate drop
  -> generic guide / defensive context noise drop
  -> risk_signal_profile
  -> quality_score / risk_score / risk_level
  -> cleaned corpus / high-risk corpus
```

## 3.3 阶段三：意图分类

### 3.3.1 Word 目标

| 项目 | 内容 |
| --- | --- |
| 阶段 | 意图分类 |
| 目标 | 将内容按风险类型（诈骗 / 引流 / 作弊 / 工具交易等）自动分类 |
| 产出物 | 风险分类模型 + 标签体系 |

### 3.3.2 当前实现方式

当前实现不是训练一个独立二进制模型，而是交付了可配置、可解释、可复跑的规则/NLP/LLM 协同分类体系：

| 文件 | 作用 |
| --- | --- |
| `src/classifier/nlp_rule_matcher.py` | fast-track 规则分类 |
| `src/enhancement/text_intelligence.py` | 细粒度意图分类、黑话/实体增强 |
| `src/enhancement/context_polarity.py` | 防御/研究/否定语境识别，降低误报 |
| `src/pipeline/classification_resolution.py` | rule / llm / final 分类仲裁 |
| `src/pipeline/stages/llm_enrich_stage.py` | 可选 LLM record enrich |
| `src/agent/model_router.py` | 判断哪些样本值得走 LLM |
| `src/agent/budget_controller.py` | LLM 调用预算控制 |
| `config/risk_taxonomy.yaml` | 风险主类、细分类、promotion marker |
| `config/label_schema.json` | 标签 schema |
| `config/model_stage_policy.yaml` | 模型阶段策略 |
| `config/routing_profiles.yaml` | fast / balanced / high_recall 预算 profile |
| `scripts/run_classification_extraction_phase.py` | 分类 / 抽取阶段脚本 |
| `scripts/evaluate_pipeline.py` | 分类效果评估 |

分类输出不只给一个标签，而是保留：

- 一级风险分类。
- 二级标签。
- 置信度。
- evidence / marker。
- `review_required`。
- 冲突状态。
- rule / llm / final 仲裁结果。

## 4.1 挑战一：数据采集与处理

### 4.1.1 Word 描述

| 全栈环节 | 核心挑战 | 对应能力 |
| --- | --- | --- |
| 数据采集与处理 | 多源异构、噪声大、重复率高 | 采集框架 + 清洗 pipeline |

### 4.1.2 当前实现

#### 多源异构

通过 `HTTPFeedCollector` 和 source catalog 统一不同来源：

- JSON / JSONL / CSV / TXT / HTML / auto 格式。
- 搜索结果页和源站页面。
- 公开网页、X、Telegram、贴吧、技术论坛、众包/二手/暗网搜索页。
- 字段归一为 `source_type / source_name / source_url / legal_basis / crawl_time / publish_time / content_text`。

1. 采集阶段 relevance：`include_keywords / exclude_keywords / include_themes / exclude_themes / min_keyword_hits`。
2. 清洗阶段 noise filter：泛教程、泛推荐、防御语境、重复内容。
3. 分类阶段 context polarity：研究、通报、反诈、否定语境不直接推高风险。

#### 重复率高

通过：

- `canonicalize_for_dedup`
- `stable_dedup_group_id`
- `DedupIndex`
- SQL/JSONL 复跑统计

## 4.3 挑战三：模型 / Agent 选型

### 4.3.1 Word 描述

| 全栈环节 | 核心挑战 | 对应能力 |
| --- | --- | --- |
| 模型 / Agent 选型 | 效果 / 成本 / 时延三角平衡 | LLM + NLP 协同、Agent 编排 |

### 4.3.2 当前实现

当前系统没有把所有 raw 数据都交给 LLM，而是采用“规则主干 + LLM 只处理高价值环节”的工程策略。

#### 规则 / NLP 主干

全量数据优先走：

```text
Clean -> Dedup -> Classify -> Extract -> Normalize -> EntityGraph -> CluePromotion
```

其中 `LLMEnrich`、`Correlate`、`Score` 是内部增强点，折叠在情报处理流水线、实体入库与线索生成中，不作为对外主流程节点。

对应代码：

- `src/pipeline/intelligence_pipeline.py`
- `src/pipeline/stages/`
- `src/classifier/nlp_rule_matcher.py`
- `src/extractor/entity_extractor.py`
- `src/enhancement/text_intelligence.py`

#### LLM 使用点

LLM 只在必要位置接入：

| 阶段 | 作用 |
| --- | --- |
| intent parse | 把用户 query 转成调查意图 |
| investigation plan | 生成 source、时间范围、证据需求计划 |
| source query rewrite | 针对 source 改写 search query |
| record enrich | 可选，但当前价值门控默认关闭 |
| clue refine | 对少量高价值线索做解释和复核增强 |

对应代码：

- `src/agent/user_request_parser.py`
- `src/agent/query_rewriter.py`
- `src/enhancement/llm_clue_refiner.py`
- `src/pipeline/stages/llm_enrich_stage.py`
- `src/backend/llm_gateway.py`

#### 预算和路由

通过以下模块控制效果 / 成本 / 时延：

| 模块 | 作用 |
| --- | --- |
| `config/routing_profiles.yaml` | 定义 `fast / balanced / high_recall` 三类预算 |
| `src/agent/budget_controller.py` | LLM 调用数、token、阶段预算控制 |
| `src/agent/model_router.py` | 判定记录或线索是否值得走 LLM |
| `src/agent/clue_ranker.py` | top clue 排序 |
| `src/agent/investigation_runtime.py` | 组装 parser、planner、retrieval、collection、processing、refine |
| `src/workflows/investigation_workflow.py` | 明确 workflow 阶段顺序和跳转原因 |

当前三类 profile：

| profile | 策略 |
| --- | --- |
| `fast` | 低延迟，默认不启用 live collection / query rewrite / record enrich，少量 clue refine |
| `balanced` | 默认交互模式，有限 source、query rewrite、record enrich 和 clue refine |
| `high_recall` | 召回优先，放宽 source、候选数、LLM 调用数和时延 |

#### 价值门控

当前消融结果显示，record enrich 多消耗 LLM 调用但没有带来已测质量收益：

| 指标 | 结果 |
| --- | --- |
| `classification_f1_delta` | 0.0 |
| `entity_f1_delta` | 0.0 |
| `hard_negative_fpr_delta` | 0.0 |
| `llm_calls_delta` | 140.0 |
| `should_enable_record_enrich` | false |
| `gate_reason` | `llm_added_cost_without_measured_quality_gain` |

所以当前工程决策是：

```text
全量主干不用 LLM 堆成本；
只在用户意图、source query rewrite、少量 clue refine 等高价值环节使用 LLM；
record enrich 只有后续证明有质量增益时才打开。
```

### 4.3.3 Agent 编排结果

当前 Agent workflow 内部仍保留 service 级 trace，便于调试和验收；对外汇报时统一折叠成 5 阶段主流程：

| --- | --- |
| 输入任务 | `query`、上传样本、指定 source、时间范围 |
| 安全与任务路由 | `run_state_preparation`、PreflightIntent、EvidenceGapGate、ConditionalPlanning、PolicyGuard、SourcePolicyGuard、routing profile、预算控制 |
| 历史资产检索 | `initial_candidate_retrieval`、`semantic_local_retrieval`、clue pool、entity graph、semantic cache |
| 情报处理流水线 | `live_collection_service`、SourceSelection、LayeredCollection、`fresh_processing_service`、Clean、Dedup、Classify、Extract、Normalize、EntityGraph |
| 线索生成与报告 | `refinement_service`、CluePromotion、RefineTargetSelector、LLMClueRefiner、ReviewRoute、`execution_summary_service`、`result_render_service` |

## 6. 安全与合规边界交付

虽然 Word 主要强调采集和分析，但黑灰产情报系统必须有边界。当前已实现：

| 边界 | 实现 |
| --- | --- |

| 生产处置默认 dry-run | `enforcement.enabled=false`、`enforcement.dry_run=true` |
| 生产处置多重门禁 | 人工审批、置信度、token、connector、allowed action/target |

这保证当前交付物用于公开/授权的防御情报分析，而不是绕过访问控制或自动化攻击。

### 7.2 明确边界

- 当前 primary 分类评测表现已通过当前 gold，但 secondary/hierarchical 仍缺正式 gold，不能声称已完成二级/层级泛化评测。
- 真实 X / Telegram 原生采集依赖外部凭据；无凭据时脚本会跳过。
- 生产处置能力只保留本地 runtime 和 dry-run 网关，不建议直接接生产执行。
- Word 中提到公众号来源；当前公开 source catalog 主要覆盖 IM/群组/论坛/社媒/垂直站点，没有把公众号作为已验证稳定来源单独交付。
