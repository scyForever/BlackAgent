## 1. 验收结论

BlackAgent 当前已经形成一套**本机可运行的黑灰产情报分析助手**。它可以把公开或已授权来源中的散乱内容，按“采集 -> 清洗 -> 分类 -> 抽取关键信息 -> 形成线索和报告”的顺序处理，帮助人工从大量帖子、群聊文本和公开页面中更快找到可复核的风险线索。

建议验收口径为：**阶段目标已达成，核心挑战已有工程化解法；同时如实保留人工确认、图片文字抽取、来源结构均衡和线上生产化边界，作为下一阶段优化项。**

需要特别说明：当前项目主形态是命令行、本机运行类和本机演示接口，不是正式对外网页服务，也不会自动封禁、自动拉黑或自动写入正式系统。

## 2. 课题要求摘要

课题要求建设一条端到端分析流程：

1. 从聊天群、群组、论坛、公开页面等来源采集黑灰产相关内容；
2. 自动去重、去噪，保留更有价值的风险内容；
3. 按诈骗引流、刷单作弊、账号交易、工具交易等风险类型进行分类；
4. 抽取黑话、账号、联系方式、链接、工具资产等关键信息；
5. 最终形成可供业务人员复核的线索、证据链和对抗建议。

启动会也明确了几条答辩口径：评判重点不是单纯堆数据量，而是流程是否完整、线索质量是否有用、分类口径是否能解释、采集是否合规、结果是否能复跑。

## 4. 核心挑战与应对情况

### 4.1 挑战一：来源多、格式乱、重复和噪声多

黑灰产信息分散在不同平台和不同表达方式里，同一条广告可能重复出现，普通教程、反诈公告、研究文章也容易混进来。当前系统的做法是：

- 先用**情报来源清单**记录来源类型、授权说明和允许访问的域名；
- 采集时保留来源、时间、访问方式等信息，后续可以追溯；
- 清洗时分开统计“原始记录”和“清洗后输入”，避免把两个口径混在一起；
- 按来源尽量均衡抽样，如果某类来源数量不足，报告里直接暴露缺口。

关键证据：

- 当前原始记录 4163 条，清洗后输入 3464 条；
- 来源结构为：聊天/群组 3786 条，社媒/论坛 356 条，垂直/技术站 21 条；
- 小规模联网试跑结果为：聊天/群组 6 条、社媒/论坛 5 条、垂直/技术站 7 条，三类都达到不少于 3 条的连通性目标；
- 清洗阶段过滤 699 条，其中重复内容 650 条、泛化教程噪声 43 条、防御语境噪声 6 条；
- 按来源尽量均衡抽样得到 1626 条，同时明确提示社媒/论坛、垂直/技术站数量不足。

结论：项目已具备合规多源采集和清洗能力；主要边界是当前原始数据仍偏聊天类来源，后续应继续补充论坛、社媒、垂直技术站等授权来源。

### 4.2 挑战二：黑话、谐音、表情和图片文字不容易看懂

黑灰产文本常故意使用黑话、谐音、缩写、表情符号，甚至把关键信息放进图片里。当前系统的做法是：

- 用规则词库识别常见黑话和变体写法；
- 判断上下文，区分黑产内容、反诈公告、研究分析和否定语境；
- 把链接、账号、联系方式统一格式，并做脱敏或哈希；
- 保留图片说明、截图引用、海报文字等字段，让后续图片文字能力可以继续增强；
- 对不确定或冲突结果打上人工复核标记。

关键证据：

- `data/collection_phase_delivery_manifest.json` 记录：谐音/变体识别 208 条，表情符号信号 186 条，图片相关文字 29 条；
- `src/enhancement/source_intake.py` 已支持图片链接、说明文字、海报字段、截图引用等进入处理流程；
- `data/eval_ocr_hardset_report.json` 显示图片文字专项测试集中，分类综合准确率为 0.900，但关键信息抽取综合准确率只有 0.4314。

结论：黑话和隐蔽表达已有可用处理链路；图片文字已经纳入输入结构，但目前不能包装成强生产能力，尤其是从图片文字中抽取账号、链接、联系方式仍是明显短板。

### 4.3 挑战三：效果、成本和速度要平衡

如果每条内容都交给大模型逐条理解，成本和速度都不可控。当前系统采用“规则优先、必要时再调用大模型”的策略：

- 简单查询、常见黑话、常见链接和联系方式先走本地规则；
- 复杂查询、联网采集规划、线索整理等高价值环节才在预算内调用大模型；
- 对逐条补充理解做价值判断，如果没有带来质量提升，就不默认开启；
- 用不同运行档位控制“更快”或“召回更多”的取舍。

关键证据：

- `data/eval_report.json`：主评测集一级分类综合准确率 1.000、二级分类 0.9524、关键信息抽取 0.9655、线索判断 1.000；离线每千条大模型调用次数为 0；95% 响应时间约 2.79 秒；
- `data/eval/latest_llm_value.json`：逐条补充理解会增加 145 次调用，但分类、关键信息、线索指标没有测得提升，因此当前策略是只处理冲突样本；
- `config/routing_profiles.yaml` 支持快速、均衡、高召回三类运行档位。

结论：当前默认策略符合“效果、成本、速度平衡”的要求，也能解释为什么不是把所有内容都直接交给大模型。

## 5. 线索质量与可复核证据链

课题答辩不能只看抽出了多少个实体，更要看线索是否能解释“为什么值得人工查”。当前系统已经把关键信息接入实体关系库、候选线索、重点线索晋升、作弊剧本和证据链展示。

| 线索类型 | 晋升条件 | 价值 |
|---|---|---|
| 同一联系方式反复出现 | 48 小时内同一联系方式至少出现 3 次 | 说明多个帖子可能由同一团伙或渠道运营 |
| 同一域名跨来源出现 | 同一域名来自至少 2 个不同来源 | 比单条帖子更可信，适合优先复核 |
| 同一话术模板重复 | 去重后同类模板至少出现 3 次 | 识别批量招募、批量推广或批量诈骗话术 |

系统输出默认只给人工复核：对抗建议不会自动封禁、不会自动拉黑、不会自动写入正式库。弱线索会归档，避免召回过多导致人工看不过来。

必须主动说明的边界：

## 7. 安全合规与运行边界

已落地控制：

- 默认不联网；只有显式打开联网参数，才会访问公开或授权来源；
- 采集器要求配置允许访问的域名和合法依据；
- 安全拦截模块会阻断正式写入、个人信息外发、未授权采集扩张和绕过限制；
- 当前 4163 条交付语料都标记为公开合规来源；
- 对抗建议默认只给人工复核，不自动触发线上处置。

运行边界：

- 当前不是正式对外网页服务，也不是多用户生产系统；
- `scripts/serve_demo_api.py` 是本机答辩演示接口，默认绑定本机地址；
- 不宣称覆盖私群、登录后页面、验证码页面或平台规则不明确页面；
- 不把本机试跑、小规模联网试跑、留出样本评测说成线上生产效果。

## 8. 建议验收演示路径

建议现场演示按“可运行入口 -> 阶段产物 -> 线索证据 -> 评测边界”顺序：

```powershell
python scripts/run_agent_cli.py --demo-sample --show clues
python scripts/serve_demo_api.py --oneshot-output data/demo_api_report.json
python scripts/export_delivery_corpora.py
python scripts/run_cleaning_phase.py
python scripts/run_classification_extraction_phase.py --source auto
python -m pytest -q
```

如现场时间有限，优先演示前两条命令：第一条展示从输入到线索输出，第二条生成本机可保存的演示报告；再用 `data` 目录下的报告解释四个阶段产物和质量边界。
