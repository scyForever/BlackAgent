    当前有 source catalog、授权采集边界、relevance 过滤、清洗去重和高危语料输出。证据：docs/交付文档.md:667-727。

  - 黑话变种 / 语义隐蔽：覆盖较好。
    代码里有黑话归一、表情/谐音变体、动态词库和上下文确认机制，例如 加V/纸飞机/接ma/群控/卡密 等：src/enhancement/
    text_intelligence.py:166-275、docs/交付文档.md:730-803。

  - 效果 / 成本 / 时延三角平衡：方向正确。
    当前不是所有样本都走 LLM，而是规则主干 + routing profile + budget controller + selective refine：config/

 ## 4. 主要不足与优化优先级

  ### P0：答辩前最值得补

  1. 补真实 held-out 评估集。
     当前测试全过，评估报告指标好，但交付文档也说明离线 gold 不能等同线上泛化：docs/交付文档.md:994。建议抽 50-100 条真
     实样本人工标注，展示混淆矩阵和典型误判。

  2. 降低人工复核负载。
     当前分类 1755 条里 review_required=739，比例偏高；答辩时容易被问“是否真的替代人工”。优先细分/校准 众包服务 / 代投服
     务 / review-only 标签。

 ### P1：增强竞争力

  3. 增强真实 OCR。
     当前有 OCR adapter 和 PBM demo，但生产 OCR 依赖 Tesseract 或外部引擎注入：src/ocr/image_text.py:28-75、src/ocr/
  4. 补跨源实体图谱展示。
     实体图谱代码已具备 cross-source clue 生成，但答辩时应展示“同一 TG/联系方式跨多个来源出现 → 线索可信度提升”的案例。

  5. 补服务器部署说明。
     启动会说本机或服务器都可以，当前本地 demo 足够 MVP；若想更像“上线”，可补 Docker 服务启动说明或只读本机 Web UI 演示。

   ### P2：长期优化

  6. 调度与监控生产化。
     已有 scheduler/queue 结构，但还需要采集失败率、源质量、重复率、token 成本、时延分布的 dashboard 化。

  7. LLM 价值门控继续做消融。
     当前策略正确：不全量调用 LLM。但要持续证明哪些环节值得用 LLM，避免“为了 Agent 而 Agent”。
