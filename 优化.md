## 2. 距离“阶段目标”还可优化的部分

### 2.1 数据采集：框架达标，但真实源验证不足

课题阶段目标要求打通 IM、群组、论坛等至少 3 类情报源，产出原始情报数据集。你的代码已经有多源配置、collector、Telegram/X/HTTP feed 相关脚本、SourcePolicyGuard 和 network 开关，作为采集框架是够的。

但还缺一份更硬的“真实采集验证报告”。

建议补：

```text
1. 至少 3 类 source 的 smoke run 报告：
   - IM / 群组类
   - 社媒 / 论坛类
   - 垂直站点 / 技术社区类

2. 每类 source 输出：
   - source_name
   - legal_basis
   - network_enabled
   - collected_count
   - filtered_count
   - duplicate_rate
   - high_risk_candidate_count
   - failure_reason

3. 明确哪些是 demo / dry-run，哪些是真实公开授权源。
```

你现在可以说：

```text
已实现多源采集框架和合规边界。
```

但不要说：

```text
已长期稳定接入所有真实黑灰产渠道。
```

---

### 2.2 智能清洗：主链路达标，但图片文字/OCR类噪声还弱

课题材料强调情报里有文本、链接、账号、图片文字，且噪声多、重复多、格式混乱。 你现在对文本清洗、去重、hard negative 做得比较好，但对“图片文字”这类场景还没有成为主链路能力。

建议下一步补：

```text
1. OCR ingestion stage：
   image → OCR text → CleanStage

2. OCR noise cleaner：
   处理错别字、断行、空格、混淆字符

3. image_text_evaluation.jsonl：
   专门测试图片文字中的联系方式、链接、黑话、工具名

4. source metadata 标记：
   content_modality = text / image_text / mixed
```

如果时间有限，至少在交付文档中说明：

```text
当前主链路以文本情报为主，图片文字 OCR 是增强项。
```

## 3. 距离“总体目标”还可优化的部分

总体目标是把散落情报自动变成结构化的作弊剧本和对抗方案。你现在已经能产出风险线索和实体库，但“作弊剧本 / 对抗方案”这一层还可以更强。

当前系统更偏：

```text
样本 → 风险分类 → 实体 → 线索
```

还可以增强为：

```text
线索 → 话术模式 → 作案链路 → 作弊剧本 / 对抗建议
```

建议新增或强化：

```text
1. PlaybookBuilder
   从多个 clue 和 entity graph 归纳：
   - 招募方式
   - 引流渠道
   - 联系方式
   - 工具资产
   - 交易/结算方式
   - 目标平台
   - 操作链路

2. CountermeasureSummary
   输出：
   - 可疑实体
   - 证据链
   - 风控关注点
   - 人工复核建议
   - 后续监控关键词

3. EvidenceChainRenderer
   每条线索必须能展示：
   - 来源
   - 原始片段
   - 命中规则
   - 抽取实体
   - 关联实体
   - 置信度
```

这样答辩时可以展示的不只是“我抽到了实体”，而是：

```text
我能把分散情报组织成一条可复核的黑灰产活动链路。
```

这会更贴近总体目标。

## 4. 距离“核心挑战”还可优化的部分

### 4.1 多源异构、噪声大、重复率高：还需要真实数据规模证明

你现在的测试集和 demo 数据能证明链路有效，但还没有证明系统在更大规模、多源真实数据上的稳定性。

建议补一个小型 benchmark：

```text
source_benchmark/
  im_samples.jsonl
  forum_samples.jsonl
  social_samples.jsonl
  vertical_site_samples.jsonl
```

每类至少：

```text
100 - 300 条样本
```

输出指标：

```text
cleaned_count
duplicate_rate
noise_rate
high_risk_candidate_rate
entity_count
clue_count
processing_latency
```

这会直接对应“多源异构、噪声大、重复率高”的挑战。

---

### 4.2 黑话变种、语义隐蔽：困难集规模仍偏小

你已经有：

```text
hard_slang_ambiguous.jsonl
context_conflict.jsonl
low_evidence_high_risk.jsonl
llm_required_cases.jsonl
cross_source_entity_graph.jsonl
slang_variants.jsonl
```

这是很好的方向。但其中部分困难集只有 2 条，更多是 smoke test，不足以证明鲁棒性。

建议优先扩：

```text
hard_slang_ambiguous: 50+
context_conflict: 50+
low_evidence_high_risk: 50+
llm_required_cases: 50+
cross_source_entity_graph: 30 组+
```

重点覆盖：

```text
谐音：接马 / 接🐴 / 飞机 / 薇
拆字：微-信 / T G / q 群
emoji：🐴 / ✈️ / 💰
规避写法：群 控 / 脚 本 / 养 号
OCR噪声：断行、错字、数字字母混淆
上下文冲突：反诈提醒、平台公告、安全研究
```

这是最贴近课题“隐蔽强”的优化点。

---

### 4.3 效果 / 成本 / 时延三角平衡：有框架，但 LLM 增益没证明

启动会明确提到不可能把上亿级别数据一条条调大模型，必须考虑效果、成本和时延，用工程化方式解决。 你的系统已经有：

```text
routing_profiles
BudgetController
LLMValueGate
ModelRouter
llm_ablation
llm_call_traces
```

这是达标的。

但 ablation 当前显示：

```text
classification_f1_delta = 0.0
entity_f1_delta = 0.0
clue_precision_delta = 0.0
clue_recall_delta = 0.0
llm_calls_delta = 145.0
should_enable_record_enrich = false
```

也就是说，当前评估集上 LLM record enrich 没有带来收益。

优化方向不是“继续多调 LLM”，而是：

```text
1. 保持默认 record_enrich 关闭或 hard_cases_only。
2. 只在以下场景启用：
   - rule/BERT 冲突
   - unknown slang
   - low confidence high risk
   - context conflict
   - complex report request

3. 在 llm_required_cases 上证明 LLM 增益。
4. 输出 tokens_per_extra_valid_clue。
```

如果你计划引入本地 BERT，可以作为 LLM 前置层：

```text
规则 → 本地 BERT → LLM only hard cases
```

这更符合成本约束。

## 5. 距离“黑灰产情报分析 Agent 课题要求”还可优化的部分

### 5.1 多轮对话机制仍不是完整形态

现在系统更像：

```text
单轮 investigation runner + 跨 run 实体/线索资产
```

还不是完整的：

```text
多轮交互式情报分析助手
```

如果答辩里用户追问：

```text
展开第 2 条线索
继续查这个 TG 号
只看 Telegram 来源再跑一次
解释为什么判成工具交易
```

你需要更明确的 conversation layer。

建议最小实现：

```text
ConversationSession
ConversationTurn
ConversationMemoryStore
FollowupParser
ConversationResolver
```

支持 5 类多轮：

```text
1. 展开第 N 条线索
2. 解释某条线索判断依据
3. 追踪某个实体
4. 修改时间/source/profile 后重跑
5. 基于当前线索生成报告
```

这不是阶段交付必需，但会明显提升 Agent 感。

---

### 5.2 Query Preflight 还可以独立化

现在 `_fallback_intent()` 承担了很多 query 解析职责，但它还不像一个正式 query parser。

建议抽出：

```text
src/query/preflight_parser.py
```

输出：

```python
PreflightIntent(
    risk_types=[],
    keywords=[],
    slang_terms=[],
    entity_types=[],
    freshness="recent",
    preferred_source_types=[],
    need_cross_source=False,
    confidence=0.0,
    needs_llm_parse=False,
)
```

这样你可以在答辩中清楚说明：

```text
LLM 不作为默认主入口；
系统先用 PreflightIntent 检索已有资产；
证据不足或 query 复杂才进入 LLM parser / fresh pipeline。
```

这会让 Agent 流程更清楚。

---

### 5.3 图谱 clue 和标准 clue 需要分层治理

当前主评估里：

```text
expected_clue_count = 2
actual_clue_count = 3
clue_precision = 0.5
clue_f1 = 0.6667
```

额外的 clue 是：

```text

actual_clue_count = 1

```

所以不是能力错，而是评估口径混了。

建议把 clue 评估拆成：

```text

  shared_contact_48h
  shared_domain_multi_source

graph_clue_eval
  entity_graph_tool_trade_cluster
  graph_shared_contact_cross_source

overall_review_load_eval
  clue_overgeneration_ratio
  review_load_per_100_records
```

这是我认为当前最应该优先优化的一项，因为它直接影响最终指标解释。

## 6. 工程交付层面的优化点

### 6.1 全量测试耗时/卡住问题

我尝试全量 pytest 但未在当前窗口完成。这可能是个别测试慢、外部依赖等待，或者测试本身没有合理 timeout。

建议：

```text
1. 给网络/集成测试打 marker：
   @pytest.mark.integration
   @pytest.mark.network

2. 默认 pytest 只跑 unit + offline。
3. integration 放单独命令。
4. 所有可能阻塞的测试设置 timeout。
```

例如：

```bash
pytest -m "not integration and not network"
pytest -m "integration"
```

这会让交付更稳。

---

### 6.2 大文件还比较多

当前仍有较大文件：

```text

src/agent/user_request_parser.py

scripts/evaluate_pipeline.py
src/local_runtime.py
src/agent/runtime_config_services.py
src/blackagent/interfaces/cli/main.py
```

这不阻塞阶段性交付，但继续生产化建议拆：

llm_gateway → provider / cache / budget / trace
evaluate_pipeline → dataset loader / metrics / renderer
CLI main → args / command / render
user_request_parser → rule parser / llm parser / schema

### 6.3 包结构还处于兼容态

目前仍然有 `src.*`、`blackagent.*`、`storage.*` 并存。交付能跑，但最终最好收敛为标准：

然后：

这属于后续工程质量优化，不是当前课题核心短板。

## 最终判断

最新版代码已经**基本达到阶段目标和总体目标的 MVP 要求**。它已经能覆盖：

```text
数据采集框架
智能清洗
一级意图分类
实体抽取
实体图谱
风险线索生成
Agent 编排
LLM/NLP 成本治理
安全边界
评估闭环
```

距离更高质量交付还有这些主要优化点：

```text
1. 图谱线索与标准线索评估口径拆分。
2. 困难样本集扩充，尤其黑话、语义隐蔽、OCR、上下文冲突。
3. LLM 增益需要在困难样本上证明，否则保持受控增强。
4. 真实多源采集需要 smoke report 支撑。
5. 二级标签需要补 gold 或明确降级为辅助字段。
6. Query Preflight 和多轮对话可以进一步增强 Agent 感。
7. 全量测试和包结构需要工程化收口。
```
