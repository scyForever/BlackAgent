# BlackAgent 采集优化点落地记录

本轮目标：按当前《优化.md》提出的采集优化清单，把“可答辩证据链、source 多样性、增量元数据、采集质量、图片文字输入、合规失败审计”落到代码、报告和测试里。边界仍保持合规采集：不绕登录、不绕验证码、不买数据、不恶意抓取。

## P0：答辩证据缺口

### 1. 采集快照口径对齐

已落地：

- `scripts/export_delivery_corpora.py` 在 manifest 里新增 `snapshot_alignment`，明确 `raw_record_count` 是权威 collection snapshot，cleaned / classification / entity 是派生视图。
- `scripts/run_classification_extraction_phase.py` 统一输出：`raw_snapshot_record_count`、`cleaned_snapshot_record_count`、`derived_input_total_count`、`phase_input_count`、`classification_count`、`entity_count`。
- 当前已重导出数据：
  - `data/collection_phase_delivery_manifest.json`: raw `4163`，cleaned `3464`。
  - `data/classification_extraction_phase_summary.json`: raw snapshot `4163`，cleaned snapshot `3464`，phase input/classification `3464`，entities `21774`。
  - `data/classification_extraction_phase_high_risk_summary.json`: raw snapshot `4163`，cleaned snapshot `3464`，high-risk phase input/classification `1061`，entities `4698`。

边界：raw 与 cleaned/classified 数字不强行相等；现在的改法是把“权威采集快照”和“过滤后的派生阶段输入”分开报告，避免把 filtered view 误说成全量 raw。

### 2. 降低 source 集中度

已落地：

- `scripts/export_delivery_corpora.py` 新增 `build_quota_sample()`，按 source class 做确定性 quota-balanced 样本，并限制单 source 占比。
- 新增产物 `data/collection_phase_quota_balanced_sample.jsonl`。
- manifest 新增：
  - `source_class_counts`
  - `source_access_type_counts`
  - `quota_balanced_sample`
  - `quota_balanced_jsonl`
- 当前 raw source class 结构：
  - `im_or_group`: `3786`
  - `social_or_forum`: `356`
  - `vertical_or_technical`: `21`
- 当前 quota sample 选中 `1626` 条，其中 IM `1249`、social/forum `356`、vertical/technical `21`。

边界：当前 raw corpus 的非 IM 记录不足，所以 quota sample 会明确给出 underfilled warning：social/forum、vertical/technical、other_authorized 无法凭空补满；这比掩盖 Telegram 集中度更适合答辩。

### 3. 补真实外部 source smoke 证据

已落地：

- `scripts/generate_source_smoke_report.py` 从“每类尝试一个代表源”改为：在合规允许前提下持续尝试配置源，直到每类达到 `min_records_per_class=3` 或源耗尽。
- smoke report 新增 `per_class_evidence`：记录每类 `configured_source_count`、`collected_count`、`target_min_records`、`target_met`、`failure_reasons`、`authorization_statements`。
- `config/intel_sources.public.yaml` 新增 `telegram_search_hub`，加强 IM/TG public smoke 覆盖。
- 当前 `data/source_external_smoke_report.json`：
  - `status=completed`
  - IM/group collected `6` / target `3`
  - social/forum collected `5` / target `3`
  - vertical/technical collected `7` / target `3`

边界：external smoke 只针对 catalog 中公开/授权、allowed domain、显式 live enabled 的来源；不代表覆盖私群、登录后页面、验证码页面或 robots/terms 不明确页面。

## P1：采集质量提升

### 4. 增量采集和去重游标

已落地：

- 新增 `src/collector/source_metadata.py`。
- `build_raw_intelligence()`、HTTP feed collector 和导出脚本都会补：
  - `content_hash`
  - `last_seen_at`
  - `last_cursor`
  - `source_snapshot_id`

边界：本轮先落字段和确定性生成逻辑；真正的跨周期 cursor 调度仍依赖后续 scheduler/collector 按这些字段做增量策略。

### 5. 采集前置质量评分

已落地：

- `src/collector/source_metadata.py` 新增 `collection_quality`，采集层就记录：
  - `slang_hit_count`
  - `contact_link_tool_hit_count`
  - `duplicate_probability`
  - `defensive_context_probability`
  - `spam_or_tutorial_noise_probability`
  - `risk_score / risk_level / risk_categories / risk_markers`
  - `quality_score`
  - `noise_reason`

边界：这是规则驱动的 pre-cleaning quality profile，不是人工标注质量分，也不是模型置信度。

### 6. 图片/OCR 文字作为一等输入

已落地：

- `src/enhancement/source_intake.py` 新增 media reference 识别：`image_url`、`poster_url`、`poster_image_url`、`screenshot_ref`、`screenshot_url`、`image_path`。
- multimodal materialize 现在输出：
  - `multimodal_reference_fields`
  - `multimodal_reference_count`
  - `content_modality=image_reference / image_text / mixed / text`
- `scripts/run_cleaning_phase.py` 会把这些 multimodal reference 字段复制到 cleaned rows。
- `scripts/export_delivery_corpora.py` 会在 raw export/manifest 样本里保留 multimodal reference 证据。

边界：这让上游 OCR/图片文本成为管线输入字段；不等于本轮新增了强生产 OCR engine。

## P2：产品化和合规审计

### 7. source provenance buckets

已落地：

- `src/collector/source_metadata.py` 新增并统一归一化：
  - `public_compliant`
  - `authorized_partner`
  - `internal_authorized`
  - `loopback_demo`
  - `manual_upload`
- manifest 输出 `source_access_type_counts`。
- 当前 delivery corpus 为 `public_compliant=4163`。

边界：provenance bucket 是证据口径字段，仍需真实 source config / legal basis 支撑，不能用字段名替代授权证明。

### 8. 失败原因和合规审计更细

已落地：

- `classify_collection_failure()` 统一归一化采集失败原因：`403`、`429`、`captcha`、`login_required`、`robots_uncertain`、`timeout`、`http_xxx`、`other`。
- `LocalAgentRuntime.collect_sources_batch()` 的 per-source result 新增 `failure_reason`。
- `scripts/generate_source_smoke_report.py` 在 live smoke 失败时记录 `failure_reason` 和 `failure_detail`，并汇总到 `per_class_evidence.failure_reasons`。

边界：失败审计用于解释为何不采某些源，不用于鼓励绕过限制。
