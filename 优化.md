## 2. 实体抽取需要修核心规则，不是只调评估口径

当前 `entity_f1 = 0.6897`，主要不是因为 pipeline 架构问题，而是实体规则和评估 gold 的口径不一致。

我看到几个典型问题。

第一，邀请码样本里：

```text
音符引流群，邀请码 code:e001，联系 13800130001
```

期望实体是：

```text
e001
```

但当前抽取结果是：

```text
account: code
```

原因是 `INVITE_CODE_RE` 对“邀请码 code:e001”这种结构会先匹配到 `邀请码 code`，然后把 `code` 当成捕获值。这个应该改成更明确的模式：

```python
INVITE_CODE_RE = re.compile(
    r"(?:邀请码|暗号|口令)[:：\\s]*(?:code[:：\\s]*)?([A-Za-z0-9_-]{3,24})"
    r"|\\bcode[:：\\s]*([A-Za-z0-9_-]{3,24})",
    re.IGNORECASE,
)
```

然后取第一个非空 group。

第二，`TG:e000` 这类评估样本没有被抽取，是因为当前 Telegram 用户名规则要求至少 5 位：

```python
[a-zA-Z][a-zA-Z0-9_]{4,31}
```

如果你的 gold 里的 `e000` 只是测试代号，那应该调整 gold；如果业务里确实会出现短代号，就要把它单独当成 `contact_alias`，不要强行当 Telegram username。

第三，URL、短链、域名、口令、邀请码、联系方式应该统一走“实体归一化层”，而不是各 extractor 各自决定最终值。

建议新增：

```text
src/intelligence/entity_normalizer.py
```

它负责：

```text
Telegram / TG / 飞机号归一
微信 / VX / 加薇归一
URL domain 提取
邀请码 code 提取
手机号标准化
账号 ID 标准化
敏感值 hash
display_value 脱敏
```

目标结构：

```python
{
  "entity_type": "contact",
  "raw_value": "TG:e001",
  "normalized_value": "telegram:e001",
  "canonical_hash": "sha256(...)",
  "masked_value": "TG:***01",
  "confidence": 0.91,
  "normalizer_version": "entity_normalizer_v1"
}
```

实体归一化做起来之后，`EntityGraphStore` 才有稳定意义。

## 7. LLM Budget 账本还有语义问题

你已经加了：

```text
BudgetLedger
attempted_calls
allowed_calls
denied_calls
consumed_tokens
denied_tokens
by_stage
```

方向是对的。但现在 `allow_llm_call()` 会记录 ledger，而部分调用方会先 pre-check，再让 `LLMGateway` 再 check 一次。

例如 `LLMEnrichStage._allow()` 里：

```python
self.budget.allow_llm_call(stage="llm_classify", estimated_tokens=0, item_count=1)
self.budget.allow_llm_call(stage="llm_classify", estimated_tokens=estimated_tokens, item_count=0)
```

随后 `LLMGateway.chat()` 又会再调用一次：

```python
budget.allow_llm_call(...)
```

`ClueRefinementService` 里也会先对每条 clue 调 `allow_llm_call()`，然后 batch refine 时 gateway 再调一次。

结果是：

```text
真实 LLM 调用次数是 1
ledger attempted_calls 可能记录成多次
allowed_calls 也会虚高
```

这会污染成本统计。

建议把预算接口改成三段式：

```python
budget.peek(stage, estimated_tokens, item_count) -> bool
budget.reserve(stage, estimated_tokens, item_count) -> BudgetLease
budget.consume(lease, ok=True, cache_hit=False)
```

调用原则：

```text
业务层只 peek 或 reserve。
LLMGateway 只 consume 已有 lease。
不要让 allow 同时承担“检查”和“记账”。
```

短期最简单的修法是：

```text
allow_llm_call 不再写 ledger；
新增 record_attempt / record_denial / consume_llm；
由 LLMGateway 统一记账。
```

这样效果 / 成本 / 时延指标才可信。

## 最终判断

这版已经比前几版成熟很多，基础架构已经不是主要短板。真正还值得优化的空间集中在六个核心方向：

```text
1. 降低误报率，尤其是公告、反诈、研究、否定语境。
2. 提升实体抽取质量，尤其是邀请码、TG、URL、联系方式归一化。
3. 让 evaluation 成为真实质量门禁，而不是 smoke test。
4. 拆掉 InvestigationRuntime 这个新的上帝类。
5. 把 pipeline 从 dict 兼容层升级成 typed domain 数据流。
6. 把 EntityGraph 从 run 内存态升级成持久化实体情报库。
```

现在不建议继续优先加新 Agent 能力。下一阶段的主线应该是：

```text
效果先达标，
评估先可信，
预算账本先准确，
runtime 先瘦身，
实体库先持久化。
```

做到这一步之后，项目就不只是“一个能跑的黑灰产分析 Agent”，而是更接近一个可治理的黑灰产情报分析系统。
