## 1. 最核心的问题：LLM 成本收益还没有被证明

以及：

```bash
  --gold tests/evaluation/gold_classification.jsonl \
  --entities-gold tests/evaluation/gold_entities.jsonl \
  --hard-negative tests/evaluation/hard_negative.jsonl \
  --clues-gold tests/evaluation/gold_clues.jsonl
```

当前离线 evaluation 指标很好看：

```text

llm_classify calls = 24
classification_f1 = 1.0
```

问题是：**LLM 调用了 24 次，但在当前评估集上没有带来可见的分类或实体指标提升**。这说明现在的 `LLMEnrichStage` 已经接入了，但它的“边际收益”还没有被验证。

这不是小问题。你的系统目标之一是效果 / 成本 / 时延三角平衡。如果 LLM 只是增加调用和 token，但不提高召回、降低误报或提升实体抽取，那它就不是增强器，而是成本中心。

建议新增一个核心模块：

```text

```

或者扩展 `scripts/evaluate_pipeline.py`，支持三组对比：

```bash
--profile fast --llm-mode off
--profile high_recall --llm-mode off
--profile high_recall --llm-mode mock
```



而是：

```text

```

建议引入一个 `LLMValueGate`：

```python
class LLMValueGate:
```

## 3. Clue 质量仍然是短板：召回够了，但过量生成还明显

当前 evaluation 里 clue 指标是：

```text

actual_clue_count = 6
clue_precision = 0.3333
clue_recall = 1.0
clue_f1 = 0.5
clue_overgeneration_ratio = 3.0
```

这说明线索生成现在偏“召回优先”，但复核负担仍然较高。对情报系统来说，clue 不是越多越好。clue 多会直接增加人工复核成本。

现在建议把 clue 生成拆成两层：

```text
candidate_clue
  宽松召回，保留更多可能线索。

actionable_clue
  满足证据链、跨源、实体敏感度、风险分阈值后才进入人工复核或告警。
```

也就是现在的 `RiskClueAggregator / EntityGraphStore.generate_clues()` 不应直接生成最终线索，而应先生成候选：

```python
class CandidateClue(BaseModel):
    clue_type: str
    key: str
    evidence_trace_ids: list[str]
    related_entity_ids: list[str]
    weak_reason: str
```

然后新增：

```text
src/pipeline/stages/clue_promotion_stage.py
```

提升规则：

```text
contact / account 类实体：
  至少 2 个 source 或 3 条 observation

domain / url 类实体：
  至少 2 个 source，且关联 contact 或 tool

tool / slang 类实体：
  仅有工具词不够，必须同时出现 contact、price、url、招募/售卖动作

template 类线索：
  至少 3 次重复，且非 defensive/research context
```

输出应分为：

```json
{
  "candidate_clues": [...],
  "actionable_clues": [...],
  "archived_weak_clues": [...]
}
```

质量门禁建议加：

```text
clue_precision >= 0.60
clue_overgeneration_ratio <= 2.0
review_load_per_100_records <= 3.0
```

现在 clue 召回已经够了，下一步应重点压过量生成。



这会让 LLM 增强更可控。

## 7. BudgetLedger 有进步，但错误分支仍可能漏记

你已经把预算控制改成：

```text

reserve
consume
BudgetLease
BudgetLedger
```

方向正确。但 `LLMGateway.chat()` 里仍有一类风险：在 `reserve()` 成功后，如果出现这些分支：

```text
missing_api_key
HTTPError
URLError
invalid_json_response
unexpected exception
```

部分路径会直接 `return response`，没有统一调用：

```python
budget.consume(lease, ok=False, network=True/False)
```

这样会导致：

```text
attempted_calls / allowed_calls 记录了
failed_calls 没有记录
consumed / failed token 账本不完整
```

建议把 `LLMGateway.chat()` 改成统一的预算完成函数：

```python
def _finish_budget(
    self,
    *,
    budget,
    lease,
    stage,
    response,
    cache_hit,
    network,
):
    if lease is not None and budget is not None and hasattr(budget, "consume"):
        budget.consume(
            lease,
            cache_hit=cache_hit,
            ok=response.ok,
            network=network,
        )
```

然后任何 return 前都调用：

```python
return self._finalize_response(...)
```

不要在每个分支里手写 budget 逻辑。

你现在 evaluation mock 路径看起来是对的，但真实 LLM 调用和异常路径才是生产成本账本最容易出问题的地方。

# 最终判断

这版已经不是“还能不能跑”的问题。它已经能跑，并且比前几版成熟很多。

现在剩余的优化空间集中在这些核心问题：

```text
1. LLM 是否真的带来效果提升，而不是只增加成本。
2. evaluation 是否真实反映 primary / secondary / clue 质量。
3. clue 是否过量生成，是否增加人工复核负担。
4. EntityGraph 是否真正成为跨 run 的结构化实体库。
5. Pipeline 是否从 dict 兼容层升级为 typed domain 数据流。
6. 规则系统是否真正配置化，而不是硬编码 + 配置并存。
7. Runtime / mixin / SQL / CLI 的复杂度是否继续收敛。
```
