## 1. 高优先级：修复 `high_recall` profile 被 fallback plan 收紧的问题

你现在的 `fast` profile 已经生效，但我测试 `high_recall` 时发现一个细节问题。

`config/routing_profiles.yaml` 中配置的是：

```yaml
high_recall:
  max_raw_records: 20000
  max_candidate_clues: 200
```

但实际运行：

```bash
python scripts/run_agent_cli.py \
  --query "找接码群控相关线索" \
  --demo-sample \
  --show json \
  --routing-profile high_recall
```

输出预算是：

```python
{
  "max_sources": 5,
  "max_raw_records": 5000,
  "max_candidate_clues": 100,
  "max_llm_refine_clues": 20,
  "max_llm_calls": 40,
  "max_llm_tokens": 50000,
  "max_elapsed_seconds": 180
}
```

原因在于 `_fallback_plan()` 里自带了：

```python
budget={
    "max_sources": 6,
    "max_raw_records": 5000,
    "max_candidate_clues": 100,
    "max_llm_refine_clues": 20,
    "max_elapsed_seconds": 180,
}
```

而 `_resolve_budget()` 里采用了“plan budget 只能收紧 profile budget”的逻辑。所以在 LLM 不可用或 mock 返回不可用 JSON 时，fallback plan 会把 `high_recall` 的 `20000 / 200` 收紧到 `5000 / 100`。

这个问题对 `fast` 没什么影响，因为 fallback plan 比 fast 更大，不会进一步放宽；但对 `high_recall` 有影响，会让高召回模式名义上存在、实际召回预算被压低。

建议改法：

```python
def _resolve_budget(...):
    raw = self._profile_budget_defaults(profile_config or {})
    plan_budget = plan.get("budget") or {}

    plan_is_fallback = (
        plan.get("llm_ok") is False
        or plan.get("llm_reason") == "fallback_plan"
    )

    if not plan_is_fallback:
        # 只有真实 LLM plan 才允许收紧 profile budget
        raw = self._apply_plan_budget_caps(raw, plan_budget)

    if policy_override is not None:
        raw = {**raw, **self._budget_override_payload(policy_override)}
```

补一个测试：

```python
def test_high_recall_profile_not_tightened_by_fallback_plan():
    orchestrator = InvestigationOrchestrator(
        llm_gateway=LLMGateway(dry_run=True, mock=True),
        routing_profiles=get_settings().routing_profiles,
    )

    result = orchestrator.run(
        "找接码群控线索",
        records=[],
        routing_profile="high_recall",
    )

    assert result.execution_summary["budget"]["max_raw_records"] == 20000
    assert result.execution_summary["budget"]["max_candidate_clues"] == 200
```

这个问题优先级高，因为它直接影响“效果 / 成本 / 时延三角平衡”的策略可信度。

## 2. 高优先级：`ModelRouter.decide_record()` 还没有真正驱动 LLM 分类 / 抽取

你现在已经有：

```python
ModelRouter.decide_record(...)
```

并且 `IntelligencePipeline.run()` 会生成：

```python
routed = [self._route_item(item) for item in extracted]
```

但目前这一步只是生成路由结果，没有真正接入：

```text
llm_classify_extract stage
```

也就是说，现在样本级协同仍然是：

```text
规则分类
实体抽取
ModelRouter 给出路由
结束
```

还不是：

```text
规则分类
实体抽取
ModelRouter 判断是否需要 LLM
高价值不确定样本进入 LLM classify / extract
合并 LLM 增强结果
```

我跑 `tests/evaluation/gold_classification.jsonl` 时就能看到这个问题：有些样本被 `ModelRouter` 判成 `llm_classify_extract`，但后面没有实际增强阶段。

建议新增一个 stage：

```text
src/pipeline/stages/llm_enrich_stage.py
```

职责：

```text
只处理 route.action == "llm_classify_extract" 的样本；
受 BudgetController 控制；
输出 enhanced_classification / enhanced_entities；
失败时保留 deterministic result；
不覆盖原始证据，只增加增强视图。
```

伪代码：

```python
class LLMEnrichStage:
    def __init__(self, llm_gateway, budget_controller, prompt_guard, output_validator):
        self.llm_gateway = llm_gateway
        self.budget = budget_controller
        self.prompt_guard = prompt_guard
        self.output_validator = output_validator

    def run_batch(self, items, routes, context):
        enriched = []

        for item, route in zip(items, routes, strict=False):
            payload = dict(item)

            if route["action"] != "llm_classify_extract":
                enriched.append(payload)
                continue

            if not self.budget.allow_llm_call(
                stage="llm_classify",
                estimated_tokens=route["max_tokens"],
                item_count=1,
            ):
                payload["llm_enrich_skipped_reason"] = "budget_denied"
                enriched.append(payload)
                continue

            response = self.llm_gateway.chat(
                messages=self._build_messages(payload, context),
                stage="llm_classify",
                budget=self.budget,
                max_tokens=route["max_tokens"],
                response_format={"type": "json_object"},
                cache_policy="read_write",
                deadline_ms=route["deadline_ms"],
            )

            payload["llm_enrichment"] = self._normalize_response(response, payload)
            enriched.append(payload)

        return enriched
```

然后 `IntelligencePipeline.run()` 应该变成：

```python
classified = self.classify_stage.run_batch(triaged, context=context)
extracted = self.extract_stage.run_batch(classified, context=context)
routed = [self._route_item(item) for item in extracted]

enriched = self.llm_enrich_stage.run_batch(
    extracted,
    routes=routed,
    context={**context, "budget": budget_controller},
)

correlated = self.correlate_stage.run_batch(enriched, routed=routed, context=stage_context)
```

这一步是你当前 LLM + NLP 协同里最缺的一环。现在 LLM 主要用于 intent、plan、query rewrite、clue refine；但对“分类 / 抽取”这个课题核心链路，还没有真正成为受控增强器。你的课题目标正是“采集 → 清洗 → 意图分类 → 实体抽取”，所以这里应该优先补。


输出大致是：

```text

entity_f1 = 0.6667
false_positive_rate = 0.6667

```

这里暴露三个问题。

第一，`unknown` 被当成了预测类别，导致 hard negative 里也可能被算成 false positive。对于评估来说，下面这些应该视为“无风险预测”：

```text

无风险

```

否则 hard negative 的 false positive rate 会被污染。

第二，实体匹配口径过严。例如期望值里有：

```text
Telegram:gold01
```

实际抽取是：

```text
gold01
```

这会被算作错误。建议实体评估改成：

```text
entity_type + normalized_value
```

而不是只比字符串。并且要有归一化函数：

```python
def normalize_eval_entity(value: str) -> str:
    text = value.lower().strip()
    text = text.replace("telegram:", "")
    text = text.replace("tg:", "")
    text = text.replace("https://", "")
    text = text.rstrip("/")
    return text
```

第三，当前 `clue_count = 0` 不一定代表 pipeline 错了，因为 `RiskClueAggregator` 的规则是：

```text
同一 contact 至少出现 3 次
同一 domain 至少跨 2 个 source
同一 template 至少出现 3 次
```

而你的 gold classification 文件只有单条样本，不适合评估 clue aggregation。建议把评估拆成三类：

```text
classification_eval
entity_eval
clue_aggregation_eval
```

不要用单条样本数据集去评估 clue 聚合。

建议改 `evaluate_pipeline.py`：

```python
NO_RISK_LABELS = {"", "unknown", "normal", "none", "无风险", "待研判"}

def predicted_categories(classification):
    risk = str(classification.get("risk_category") or "").strip()
    secondary = str(classification.get("secondary_label") or "").strip()

    labels = set()
    if risk.lower() not in NO_RISK_LABELS:
        labels.add(risk)
    if secondary and secondary not in {"待研判", "未细分"}:
        labels.add(secondary)
    return labels
```

再新增一个 clue 聚合评估集：

```text
tests/evaluation/gold_clues.jsonl
```

里面至少放 3 条共享 contact、2 条跨 source domain、3 条模板重复样本，否则 clue 聚合没有触发条件。

## 4. 高优先级：LLM Gateway 统计是累计的，不是单次 run 的

当前 `LLMGateway.stats()` 会返回 gateway 实例生命周期内的全部调用。`RuntimeContainer` 会复用同一个 gateway，所以连续跑两次 investigation 时，第二次的 telemetry 会包含第一次的 LLM 调用。

我用同一个 `LocalAgentRuntime` 连续跑两次 balanced investigation，第二次的 `llm_gateway.call_count` 变成了 6，而不是单次 run 的 3。这会污染单次任务的成本统计。

建议加两种能力之一。

方案 A：在 run 开始时记录 offset：

```python
stats_start_index = len(self.llm_gateway.stats())
...
run_stats = self.llm_gateway.stats()[stats_start_index:]
```

方案 B：给 gateway 增加 run-scoped stats：

```python
class LLMGateway:
    def stats_count(self) -> int:
        return len(self._stats)

    def stats_since(self, index: int) -> list[dict[str, Any]]:
        return [item.model_dump() for item in self._stats[index:]]
```

然后在 `_prepare_run_state` 里记录：

```python
gateway_stats_start = self.llm_gateway.stats_count()
```

最后 summary 使用：

```python
llm_gateway_stats = self.llm_gateway.stats_since(run_state.gateway_stats_start)
```

不要在每次 run 前直接 `clear_stats()`，因为那会影响并发或外部观测；用 offset 更稳。

## 5. 高优先级：LLM cache key 仍然太依赖动态字段

你已经给 `LLMGateway` 加了进程内 cache，并且 intent_parse / investigation_plan 能命中 cache。但我连续跑相同三条样本时，`clue_refine` 没有 cache hit。

主要原因是 `clue_refine` 的 prompt 里包含完整 clue dict，而 clue dict 里有动态字段，例如：

```text
clue_id
created_at
orchestration_origin
quality
refine_priority_score
```

每次生成 clue 时 `clue_id` 不同，所以 cache key 也不同。

建议给 LLMGateway 支持显式 cache key：

```python
response = llm_gateway.chat(
    messages=messages,
    stage="clue_refine",
    cache_policy="read_write",
    cache_key_hint=stable_refine_cache_key(clues),
)
```

稳定 key 不应该包含 clue_id / created_at，而应该基于：

```text
stage
model
prompt_version
risk_category
clue_type
key
normalized entity_values
evidence text hash
source_type set
```

例如：

```python
def stable_clue_refine_key(clues):
    canonical = [
        {
            "clue_type": clue.get("clue_type"),
            "key": clue.get("key"),
            "risk_category": clue.get("risk_category"),
            "entity_values": sorted(clue.get("entity_values") or []),
            "source_types": sorted(clue.get("source_types") or []),
            "evidence_trace_ids_hash": hash_tuple(clue.get("evidence_trace_ids") or []),
        }
        for clue in clues
    ]
    return sha256_json({"stage": "clue_refine", "items": canonical})
```

再进一步，可以增加持久化 cache：

```text
storage/llm_cache_repo.py
```

但在加持久化前，先把 cache key 变稳定，否则持久化也命中不了。

## 6. 高优先级：发送给 LLM 的 source / clue 内容还需要脱敏和瘦身

你现在已经用了 `PromptGuard` 包裹 query 和 clue，这是对的。但 `PromptGuard` 只是在语义上声明“这是不可信数据”，它不负责脱敏，也不负责字段裁剪。

当前 `LLMSourceQueryRewriter.rewrite()` 里会把完整 `source_payload` 放进 prompt：

```python
f"source={source_payload}\n"
```

这有潜在问题。source 里未来可能包含：

```text
headers
cookies
tokens
internal source_url
allowed_domains
auth metadata
```

不应该原样发送给外部 LLM。

建议做一个 source prompt sanitizer：

```python
def sanitize_source_for_llm(source):
    return {
        "source_name": source.get("source_name"),
        "source_type": source.get("source_type"),
        "query_theme": source.get("query_theme"),
        "search_query": source.get("search_query"),
        "query_term": source.get("query_term"),
        "has_query_template": bool(source.get("query_url_template")),
    }
```

明确禁止传：

```text
headers
authorization
cookie
webhook
token
api_key
source_url 完整值
internal_ref
```

同理，`LLMClueRefiner.refine_batch()` 不建议把完整 clue 传给 LLM。应构造一个最小 clue card：

```python
def sanitize_clue_for_llm(clue):
    return {
        "clue_id": clue.get("clue_id"),
        "clue_type": clue.get("clue_type"),
        "risk_category": clue.get("risk_category"),
        "quality_score": clue.get("quality_score"),
        "confidence": clue.get("confidence"),
        "source_count": len(clue.get("source_names") or []),
        "evidence_count": len(clue.get("evidence_trace_ids") or []),
        "entity_types": clue.get("entity_types"),
        "masked_entity_values": mask_entities(clue.get("entity_values") or []),
        "threshold_reason": clue.get("threshold_reason"),
    }
```

原则是：

```text
LLM 需要知道“为什么这是线索”，不需要知道完整账号、完整联系方式、完整 URL、完整内部字段。
```

## 8. 中高优先级：`IntelligencePipeline` 已接真实组件，但主链路仍有 fallback 到 `PhaseTwoThreeEngine`

这意味着真实主链路仍然有两套：

```python
payload["pipeline_backend"] = "intelligence_pipeline"
```

## 12. 中优先级：source 合规校验应前移到 Orchestrator

`HTTPFeedCollector` 里已经做了 source authorization 检查，这是好的：

```text
legal_basis
allowed_domains
forbidden markers
credential in URL
```

但 `InvestigationOrchestrator._select_sources()` 本身没有过滤 source legal basis。如果调用方传入自定义 `collect_source_records`，就可能绕过 `HTTPFeedCollector` 的边界。

建议在 Orchestrator 侧加一层 `SourcePolicyGuard`：

```python
AUTHORIZED_LEGAL_BASES = {
    "AUTHORIZED_PARTNER",
    "PUBLIC_COMPLIANT_DATA",
    "INTERNAL_AUTHORIZED_SOURCE",
    "THIRD_PARTY_AUTHORIZED_FEED",
}

def validate_source_for_collection(source):
    legal_basis = str(source.get("legal_basis") or "")
    if legal_basis not in AUTHORIZED_LEGAL_BASES:
        raise SafetyPolicyViolation("missing_authorized_legal_basis")

    if source.get("allow_login_bypass") is True:
        raise SafetyPolicyViolation("login_bypass_forbidden")

    if source.get("allow_interaction") is True:
        raise SafetyPolicyViolation("interaction_forbidden")
```

调用点应该在：

```python
_run_live_collection_phase()
```

真正 collect 前执行，而不是只依赖 collector 实现。

## 13. 中优先级：PII masking 还比较窄

现在 `PIIMasker` 主要处理：

并区分两种输出：

```text
internal_view:
```

现在 `_mask_execution_summary()` 会 mask summary，但 `--show json` 仍可能输出 clue 里的原始 `entity_values`。后续建议所有用户可见输出走统一 renderer，而不是直接 dump 内部对象。
