有。现在的优化建议不应该再围绕“补功能”，而应该围绕 **交付质量、真实效果、流程效率和下一阶段可演进性**。你现在的代码已经能支撑黑灰产情报分析 Agent 的 MVP 交付，但如果继续优化，建议优先动下面这些核心点。

你的课题目标是把多源黑灰产情报自动完成“采集 → 清洗 → 意图分类 → 实体抽取”，并产出风险线索、风险样本和结构化实体库；核心挑战包括多源异构、噪声重复、黑话变种，以及效果 / 成本 / 时延平衡。后续优化也应该围绕这几个目标，而不是继续堆 Agent 功能。

## 1. 优先优化黑话变种识别，这是当前最贴近课题难点的短板

主评估表现很好，但单独看 `slang_variants.jsonl`，当前结果仍然偏弱：

```text
primary_classification_f1 = 0.5
entity_f1 = 0.7333
```

这说明系统对普通样本、规则友好样本已经够用，但对黑话、谐音、拆字、emoji、规避写法的鲁棒性还不够。这个问题非常重要，因为课题背景里明确提到黑灰产情报“隐蔽强”，黑话变种是核心挑战。

建议新增一个专门的 `SlangVariantNormalizer`，不要只靠词典匹配。

目标能力：

```text
薇 / 微 / V / vx / +v → 微信
飞机 / 纸飞机 / TG / T G → Telegram
接🐴 / 接马 / jm / 接 ma → 接码
裙 / 羣 / q群 / 群 → 群组
料 / 数据 / 库 / 客资 → 数据资源
```

建议实现三层：

```text
第一层：字符归一化
  繁简、大小写、emoji、空格、符号、同音替换。

第二层：黑话候选召回
  规则词典 + 拼音相似 + 编辑距离 + embedding 近邻。

第三层：上下文确认
  只有出现在招募、交易、引流、联系方式附近时，才提升风险。
```

不要把黑话词一命中就判风险，否则会误报。更好的判断是：

```text
黑话词 + 联系方式
黑话词 + 价格
黑话词 + 出售 / 接单 / 拉群 / 详聊
黑话词 + URL / 口令 / 邀请码
```

这会直接提升 `slang_variants` 的 F1，也是最贴近课题价值的优化。

## 2. 让 LLMValueGate 真的控制运行时，不要只停留在评估报告

你现在已经有 LLM ablation，结果很明确：在当前评估集上，LLM record enrich 增加了调用成本，但没有带来明显指标增益：

```text
classification_f1_delta = 0.0
entity_f1_delta = 0.0
clue_precision_delta = 0.0
clue_recall_delta = 0.0
llm_calls_delta = 140.0
should_enable_record_enrich = false
```

这说明 LLM 的使用策略还可以继续收紧。下一步不是继续调 prompt，而是让 ablation 的结论真正影响运行时。

建议你把流程改成：

```text
没有 latest_llm_value.json：
  fast：关闭 record_enrich
  balanced：只允许 hard cases
  high_recall：只允许 conflict / slang / low-confidence cases

latest_llm_value.json 显示 LLM 无收益：
  record_enrich = conflict_only

latest_llm_value.json 显示 LLM 在某类样本有收益：
  只对该类样本开启 LLM
```

运行时策略可以这样设计：

```python
if value_report is None:
    record_enrich_policy = "hard_cases_only"
elif value_report["should_enable_record_enrich"] is False:
    record_enrich_policy = "conflict_only"
else:
    record_enrich_policy = "enabled"
```

再把 `ModelRouter` 的判断从：

```text
高风险 + 低置信 → 进 LLM
```

改成：

```text
高风险 + 低置信 + LLMValueGate 允许 → 进 LLM
```

这样你的系统才能真正体现“LLM + NLP 协同”，而不是“LLM 可用就调用”。

## 3. Agent 流程建议改成 EvidenceGap 驱动

现在流程已经比较完整，但还可以进一步优化顺序。理想流程不是“先规划，再采集，再分析”，而是：

```text
先查已有资产，看缺什么证据，再决定要不要规划和采集。
```

建议把主流程调整为：

```text
Request
  ↓
PolicyGuard
  ↓
PreflightIntent
  不调用 LLM，只做轻量 query 解析
  ↓
Retrieval Fanout
  ├── clue pool
  ├── entity graph
  └── semantic local cache
  ↓
EvidenceGapGate
  判断是否已有足够证据
  ↓
如果足够：
  promotion → selective refine → report

如果不足：
  conditional planning → source selection → layered collection → fresh pipeline
```

核心新增对象可以是：

```python
class EvidenceGap(BaseModel):
    need_more_samples: bool = False
    need_recent_signals: bool = False
    need_cross_source_support: bool = False
    need_entity_chain: bool = False
    missing_entity_types: list[str] = []
    preferred_source_types: list[str] = []
    current_high_quality_count: int = 0
    required_high_quality_count: int = 0
    reasons: list[str] = []
```

这样 source 选择就不是泛泛地“继续采集”，而是更精确地回答：

```text
缺联系方式？
缺跨源证据？
缺最近样本？
缺 URL / domain？
缺实体链路？
```

这会同时优化成本、时延和解释性。

## 4. EntityGraph 应该参与前置决策，而不是只做后处理

你现在已经有实体图谱和 SQLite 持久化，这是很好的基础。但下一步应把 EntityGraph 从“pipeline 的产物”升级为“Agent 决策资产”。

当前更像是：

```text
fresh records
  ↓
extract entities
  ↓
write graph
  ↓
generate graph clues
```

建议升级为：

```text
query
  ↓
preflight intent
  ↓
entity graph retrieval
  ↓
evidence gap
  ↓
决定是否需要采集
```

举例：

```text
用户问：找群控工具交易线索

系统先查：
  tool_name=群控
  related contact
  related domain
  recent observations
  cross-source entities

如果已有：
  contact + tool + domain + 多 source
则直接生成图谱线索，不必重新采集。
```

建议新增：

```text
EntityGraphRetrievalService
EntityRiskProfileService
GraphClueGenerator
```

实体风险画像可以类似：

```json
{
  "entity_id": "ent_xxx",
  "entity_type": "contact",
  "risk_categories": {
    "工具交易": 12,
    "诈骗引流": 4
  },
  "source_count": 5,
  "observation_count": 28,
  "related_entity_count": 13,
  "risk_score": 87,
  "first_seen": "...",
  "last_seen": "..."
}
```

这会让你的系统从“样本分析器”进一步升级为“情报资产系统”。

## 5. 二级标签要么补齐评估，要么降级为辅助字段

目前一级分类已经可以交付，但二级标签还没有形成完整闭环。你的评估里二级分类是：

```text
secondary_classification_status = not_applicable
hierarchical_classification_status = not_applicable
```

这不是代码错误，而是 gold 数据没有完整二级标签。下一步必须做一个选择：

```text
方案 A：交付时只承诺一级分类。
  二级标签作为辅助研判字段，不纳入正式指标。

方案 B：正式做层级分类。
  补 expected_secondary_labels。
  输出 secondary_f1 / hierarchical_f1。
  对二级标签加置信门控。
```

如果选 B，建议把结构改成：

```json
{
  "primary": {
    "label": "工具交易",
    "confidence": 0.91
  },
  "secondary": {
    "final": null,
    "candidates": [
      {
        "label": "群控脚本",
        "confidence": 0.62,
        "evidence": ["群控", "脚本"]
      }
    ]
  }
}
```

二级标签进入 `final` 的条件应该更严格：

```text
至少 2 个二级特征词；
或 1 个二级特征词 + 关键实体；
或 LLM 与规则一致；
或历史人工复核确认。
```

否则二级标签容易变成噪声源。

## 6. 继续推进 typed-first pipeline，减少 legacy dict 兼容层

你现在已经有 `PipelineItem` 和 typed contract，方向是对的。但系统里仍然存在不少 legacy dict 快照和兼容字段。

下一步建议把 pipeline 内部彻底收敛成：

```text
PipelineItem 是唯一内部数据结构
dict 只用于 CLI / JSON 输出
```

目标结构：

```python
class PipelineResult(BaseModel):
    items: list[PipelineItem]
    candidate_clues: list[CandidateClue]
    actionable_clues: list[ActionableClue]
    archived_weak_clues: list[ArchivedWeakClue]
    execution_summary: PipelineExecutionSummary

    def to_legacy_dict(self) -> dict:
        ...
```

每个 stage 的接口固定为：

```python
def run_batch(
    self,
    items: list[PipelineItem],
    context: PipelineContext,
) -> list[PipelineItem]:
    ...
```

好处是：

```text
classification 不会被多个 stage 随意覆盖；
rule / llm / final / resolution 边界清楚；
entity 字段不漂移；
evaluation 和 runtime 读取同一份结构；
后续接 API / UI 更容易。
```

这属于工程质量优化，不一定影响当前演示效果，但会显著降低后续维护成本。

## 7. 规则系统继续向“配置唯一来源”收敛

现在已经有 `RuleRegistry`，但仍可能存在部分硬编码规则。下一步建议明确原则：

```text
规则内容必须来自配置；
Python 只保留规则执行器。
```

配置应覆盖：

```text
风险标签
二级标签
黑话归一化
实体正则
防御语境
clue promotion 条件
ModelRouter 阈值
```

建议把 clue promotion 也完全配置化，例如：

```yaml
clue_promotion:
  shared_contact_48h:
    require_any:
      - min_sources: 2
      - min_observations: 3
    require_entity_types:
      - contact
    actionability_bonus:
      sensitive_entity: 0.18

  tool_trade_cluster:
    require_all:
      - entity_type: tool_name
      - any_related_entity_type: [contact, url, domain]
      - min_observations: 2
```

交付时你可以说：

```text
新增风险规则、黑话、实体模式、线索提升规则，不需要改代码，只改配置。
```

这会比“代码里写了很多规则”更有工程说服力。

## 8. Live Collection 可以改成分层采集 + 中途停止

当前有分层采集能力，但还可以更精细。建议每采集完一层就重新评估 `EvidenceGap`：

```text
Layer 1：高可信本地 / 授权源
  ↓
EvidenceGap Recheck
  满足则停止

Layer 2：公开论坛 / 社媒
  ↓
EvidenceGap Recheck
  满足则停止

Layer 3：低质量噪声源
  ↓
最后补充
```

不要让 high_recall 直接向最大采集量靠近。真正合理的策略是：

```text
最小足够证据原则
```

也就是：

```text
有足够高质量 clue，就不要继续扩大采集；
有跨源证据，就不要继续采低质量源；
有实体链路，就不要继续重复抓同类样本。
```

这会显著降低时延和噪声。

## 9. 增加“困难样本”评估集，验证 LLM 真实价值

目前主评估表现很好，但 LLM ablation 没看到收益。可能有两个原因：

```text
1. LLM 真的没有收益；
2. 当前评估集太规则友好，没有覆盖 LLM 擅长的困难样本。
```

建议新增或扩充这些评估集：

```text
hard_slang_ambiguous.jsonl
  黑话、谐音、拆字、emoji、OCR 断裂文本

context_conflict.jsonl
  同时出现交易词和治理词，例如“平台公告曝光群控脚本交易”

low_evidence_high_risk.jsonl
  文本很短，但包含联系方式、短链、口令

cross_source_entity_graph.jsonl
  单条样本无风险，跨 source 聚合后风险升高

llm_required_cases.jsonl
  规则应失败，LLM 应能通过上下文修正
```

然后 ablation 分子集输出：

```text
slang_f1_delta
context_conflict_f1_delta
low_evidence_recall_delta
graph_clue_recall_delta
tokens_per_extra_valid_clue
```

这样你才能证明：

```text
LLM 不是所有地方都用，而是在困难样本上有用。
```

## 10. 交付层面建议补一个“系统边界说明”

你现在功能比较多，交付时容易被问：

```text
能不能真实采 Telegram？
能不能处理暗网？
能不能自动封禁？
能不能自动访问黑链？
能不能识别所有黑话？
```

建议补一份 `docs/boundary_and_limitations.md`，明确写：

```text
允许：
  授权数据源采集
  公开数据分析
  文本清洗
  风险分类
  实体抽取
  线索生成
  人工复核辅助

禁止：
  绕过登录 / 验证码 / 访问控制
  自动加入私密群
  与黑灰产账号互动
  下载或运行未知工具
  自动购买样本
  自动处置真实对象
```

再写清楚当前限制：

```text
一级分类已验证；
二级标签为辅助字段；
黑话变种仍需扩充；
图谱推理为增强方向；
真实采集源稳定性需按环境验证；
LLM 增益需结合困难样本评估。
```

这会让交付更稳。

# 建议的下一轮优先级

按收益排序，我建议你下一步做这 5 件事：

```text
1. 黑话变种识别专项优化
   目标：slang_variants primary F1 从 0.5 提到 0.8+。

2. LLMValueGate 接入运行时默认策略
   目标：没有证明收益的场景不默认启用 record_enrich。

3. Preflight-first + EvidenceGap 流程改造
   目标：先查资产，再决定是否规划和采集。

4. EntityGraph 前置检索和风险画像
   目标：结构化实体库不只是存储，而是参与决策。

5. 二级标签体系决策
   目标：要么只承诺一级分类，要么补二级 gold 并做正式指标。
```

最终判断：现在项目已经达到 MVP 交付水准；继续优化的最大价值不在“更多 Agent 能力”，而在 **黑话鲁棒性、LLM 成本收益闭环、证据缺口驱动流程、实体图谱前置决策和评估体系可信度**。
