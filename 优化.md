 主要差距

  1. 分类效果还不足以支撑“严格符合”
     当前规则和模型在固定评估集上很好，但 manual heldout 指标明显下降：主类 F1 0.7632、二级 F1 0.6102、误报率 0.2773，说
     明泛化能力和二级标签边界还不稳。课题核心是“高风险意图分类”，这里需要继续打磨。

  2. 真实多源采集的说服力还不够强
     代码层面已经覆盖多源，但配置默认 network.enabled=false，现有 live smoke 更偏验证采集框架，而不是持续真实外部源采
     集。全量数据也存在来源不均衡问题，IM/群组样本占比高，垂类/技术社区样本偏少。

  3. LLM Agent 价值还没有被结果证明
     项目有 Agent 编排、预算控制、路由和值门控，但报告里 llm_added_cost_without_measured_quality_gain 说明当前真实收益未
     证明。答辩时应强调“规则优先、LLM 只在高价值环节介入”，而不是把 LLM 作为主要质量来源。

  4. 产品化闭环偏弱
     当前更像高质量 pipeline + CLI/demo，而不是完整运营工作台。课题虽允许本机或服务器部署，但“全栈挑战赛”语境下，最好补
     一个可审阅、可导出、可追踪的轻量 Web 界面。

  优化建议优先级

  1. 优先提升分类泛化能力
     针对 manual heldout 做错误分析，把误报、漏报、二级混淆拆成单独 buckets。目标建议：主类 F1 提到 0.85+，二级 F1 提到
     0.75+，误报率压到 0.15 以下，人工复核量降到每百条 20 以下。

  2. 补强真实多源采集证据
     增加合规授权的小规模真实采集样本，按 IM/群组、公众号/文章、社交论坛、垂类/技术社区做平衡 quota。每条保留
     source_url、crawl_time、raw_payload_uri、快照、清洗原因和实体来源片段。

  3. 做 query-level 评估，而不只做 record-level 评估
     启动会提到用户可能问“取当天诈骗引流线索信息”。建议新增查询级基准：输入自然语言任务，评估返回线索的 Top-K 准确率、证
     据完整度、延迟、人工可复核性。

  4. 补轻量审阅工作台
     做一个最小 Web UI：线索列表、证据片段、实体高亮、分类结果、人工确认/驳回、导出报告。已有 storage、review repo 和
     pipeline 基础，补 UI 的收益会很明显。

  5. 强化 OCR 与图片情报链路
     接入真实 OCR 引擎或明确可插拔实现，补真实截图/海报/二维码/聊天图片样本，输出 OCR 文本和原图证据绑定关系。
