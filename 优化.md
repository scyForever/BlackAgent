有。你这版的模块已经比较完整，但 **Agent 主流程仍然有优化空间**。现在的问题不再是“有没有清洗、分类、抽取、图谱、LLM 路由”，而是这些能力在流程中的顺序、触发条件和停止条件还可以继续收紧。

我看了这版 `BlackAgent-dev(5).zip`，并跑了几个关键验证：

```text

targeted tests：test_run_agent_cli / test_evaluate_pipeline / test_refactor_boundaries 共 40 个通过
evaluate_pipeline 可运行
ablation 可运行
```

全量 `pytest` 在当前环境里超过超时窗口，未完整返回，所以我不把“全量测试通过”作为结论。下面只基于我实际跑通和检查过的流程说。

你的课题目标是把多源黑灰产情报走完“采集 → 清洗 → 意图分类 → 实体抽取”，最终产出风险线索、风险样本和结构化实体库；同时核心挑战是多源异构、噪声重复、黑话隐蔽，以及效果 / 成本 / 时延三角平衡。 当前代码已经覆盖这些能力，但流程还有进一步优化空间。

# 一、当前 Agent 主流程大致是这样

从代码看，现在主流程是：

```text
InvestigationWorkflow.run
  ↓
RunStatePreparationService
  - 解析 intent
  - 生成 investigation plan
  - 解析 routing profile
  - 生成 budget
  - 选择 source
  ↓
InitialCandidateRetrievalService
  - 从 clue pool 检索候选线索
  - 汇总 pool 质量
  ↓
SemanticLocalRetrieval
  - 条件满足时做本地语义召回
  - 扩展 graph neighbors
  ↓
LiveCollection
  - 根据 pool 质量和 source policy 决定是否实时采集
  - query rewrite
  - source collection
  ↓
FreshProcessing
  - OfflineClueBuilder
  - IntelligencePipeline
      Clean
      Dedup
      Classify
      Extract
      Route
      LLMEnrich
      Correlate
      CluePromotion
      Score
  ↓
ClueRefinement
  - merge pool clues + fresh clues
  - rank
  - ModelRouter 判断是否 refine
  - LLM batch refine
  ↓
ExplorationHypothesis
  ↓
ExecutionSummary / ResultRender
```

这个流程已经比早期版本合理很多，但现在仍然有几个核心可优化点。

# 二、最大流程优化点：先检索，再规划

当前流程里，`RunStatePreparationService` 会先做 intent parse 和 investigation plan，然后才进入 clue pool / entity graph / semantic local 检索。

也就是说，当前顺序大致是：

```text
用户 query
  ↓
intent parse / plan
  ↓
clue pool retrieval
  ↓
semantic local / live collection
```

问题是：很多查询其实可以直接通过已有 clue pool、实体图谱或本地语义库回答，不一定需要先进入完整 plan。尤其是 `balanced` 和 `high_recall` 模式下，intent / plan 可能走 LLM 或 mock LLM；即使后面发现已有线索足够，前面的规划成本也已经花掉了。

建议改成 **Preflight-first**：

```text
用户 query
  ↓
PolicyGuard
  ↓
轻量 PreflightIntent
  - 只用规则解析风险词、时间范围、source 偏好
  - 不调用 LLM
  ↓
并行/顺序检索：
  - clue pool
  - entity graph
  - local semantic cache
  ↓
EvidenceGap Gate
  ↓
如果已有证据足够：
    直接进入 promotion / selective refine / report
否则：
    再进入 LLM intent parse / investigation plan / source selection / live collection
```

推荐新流程：

```text
Request
  ↓
Safety Check
  ↓
Cheap Preflight Intent
  ↓
Retrieval Fanout
  ├── Clue Pool
  ├── Entity Graph
  └── Semantic Local Cache
  ↓
EvidenceGapDecision
  ├── sufficient → refine/report
  └── insufficient → LLM plan + source collection + fresh pipeline
```

这样可以把 LLM planning 从“默认前置步骤”变成“证据不足时才启用的增强步骤”。

这是当前流程最值得改的地方。

# 三、第二个优化点：Semantic Local 不应该依赖 Live Collection 决策

现在 `_run_semantic_local_phase()` 里的逻辑是：

```python
should_collect_live, live_collection_reasons = self._initial_live_collection_decision(...)
...
if not (should_collect_live and not retrieval_state.provided_records):
    return _SemanticLocalState(...)
```

这意味着：**只有当系统认为需要 live collection 时，才会尝试 semantic local retrieval**。

这个耦合不太合理。

Semantic local / graph expansion 是低成本本地检索，应该位于 clue pool 之后、live collection 之前。尤其在 `fast` 模式下：

```yaml
enable_live_collection: false
```

这时 live collection 被禁用，semantic local 也容易被跳过。但 fast 模式恰恰应该优先使用：

```text
clue pool
entity graph
semantic local cache
```

而不是只查 pool 后直接结束。

建议把 retrieval 层拆成三级：

```text
Tier 1: Clue Pool Retrieval
  查已有高质量线索

Tier 2: Entity Graph / Semantic Local Retrieval
  查本地实体图谱、向量缓存、历史样本

Tier 3: Live Collection
  只有前两层不足，并且 source policy 允许，才实时采集
```

改造后的决策顺序：

```text
clue_pool_summary
  ↓
如果不足
  ↓
entity_graph + semantic_local
  ↓
如果仍不足
  ↓
live collection
```

不要让 semantic local 依赖 `should_collect_live=True`。

# 四、第三个优化点：把“是否继续采集”改成 EvidenceGap 对象

现在 live collection 决策主要通过这些 reason：

```text
no_candidate_clues_in_pool
insufficient_high_quality_pool_clues
need_fresh_signals_for_short_time_window
need_recent_high_quality_signals
insufficient_cross_source_support
evidence_chain_not_satisfied_by_pool
```

这些 reason 已经有价值，但它们只是字符串，后续 source selection、collection、LLM refine 很难精确利用。

建议引入一个结构化对象：

```python
class EvidenceGap(BaseModel):
    need_more_samples: bool = False
    need_recent_signals: bool = False
    need_cross_source_support: bool = False
    need_entity_chain: bool = False
    need_contact_or_url: bool = False
    need_specific_source_types: list[str] = []
    missing_entity_types: list[str] = []
    current_high_quality_count: int = 0
    required_high_quality_count: int = 0
    reasons: list[str] = []
```

然后流程决策不再是：

```text
是否 live collection: true / false
```

而是：

```text
还缺什么证据？
```

例如：

```json
{
  "need_more_samples": true,
  "need_cross_source_support": true,
  "missing_entity_types": ["contact", "domain"],
  "need_specific_source_types": ["telegram", "forum"],
  "reasons": [
    "insufficient_cross_source_support",
    "evidence_chain_not_satisfied_by_pool"
  ]
}
```

这样 source selector 可以更精准：

```text
缺 cross-source → 优先选择不同 source_type
缺 contact → 选择联系方式密度高的数据源
缺 recent → 选择实时源
缺 entity chain → 优先实体图谱扩展
```

这会把当前的“经验式流程”升级成“证据缺口驱动流程”。

# 五、第四个优化点：Promotion 应该早于 LLM refine

现在 pipeline 内部已经有：

```text
CandidateClue
ActionableClue
ArchivedWeakClue
CluePromotionStage
```

这是正确的。但 runtime 层的 refine 仍然是对合并后的 candidate 集合做 rank 和 refine：

```text
pool clues + semantic clues + fresh clues
  ↓
merge
  ↓
rank
  ↓
ModelRouter decide_clue_refinement
  ↓
LLM refine
```

从 high_recall demo 看，当前会出现：

```text
merged_candidate_count = 9
clue_refine batch = 9 item traces
high_quality_clues = 2
candidate_clues = 7
```

这说明 refine 仍然会覆盖不少最终没有进入 high quality 的候选。

更好的流程是：

```text
candidate clues
  ↓
Promotion Gate
  ├── actionable clues
  ├── near-threshold clues
  └── archived weak clues
  ↓
只对 actionable / near-threshold 做 LLM refine
```

也就是说，LLM refine 的目标不应是“所有候选线索”，而应是：

```text
1. 已经 actionable，但摘要/证据组织不足
2. 接近 actionable 阈值，LLM 可辅助解释
3. 高风险但规则证据冲突，需要人工复核辅助
```

不应该 refine：

```text
1. archived weak clues
2. risk_category=unknown 且无敏感实体的 graph clues
3. 单源、单实体、无交易动作的弱线索
4. 已经 high_quality 且证据链完整的线索
```

建议流程改成：

```text
Correlate
  ↓
CandidateClue
  ↓
CluePromotionStage
  ↓
RefineTargetSelector
  ├── actionable_missing_summary
  ├── near_threshold_high_risk
  └── conflict_review
  ↓
LLM batch refine
  ↓
Final Review Routing
```

这会显著降低 clue_refine 的 token 消耗。

# 六、第五个优化点：LLM ValueGate 应该变成流程开关，不只是 evaluation 结果

你已经有 `LLMValueGate` 和 ablation。当前 ablation 结果很关键：

```text
high_recall_mock 相比 high_recall_off：
classification_f1_delta = 0.0
entity_f1_delta = 0.0
clue_precision_delta = 0.0
clue_recall_delta = 0.0
llm_calls_delta = 140.0
gate_reason = llm_added_cost_without_measured_quality_gain
should_enable_record_enrich = False
```

这说明在当前评估集上，record-level LLM enrich 没有带来可见收益。

现在问题是：流程里 `LLMValueGate` 主要依赖 `data/eval/latest_llm_value.json`。我在包里没有看到这个文件默认存在：

```text
data/eval/latest_llm_value.json 不存在
```

所以 runtime 默认可能仍然启用 record enrich，除非你手工写入这个报告。

建议把流程改成“收益未知时保守”：

```text
如果 latest_llm_value.json 不存在：
  fast: record_enrich = false
  balanced: 只允许 conflict hard case
  high_recall: 只允许 slang/context_conflict/low_confidence 子集
```

而不是：

```text
没有 value report → 默认启用 record_enrich
```

推荐逻辑：

```python
if value_report is None:
    record_enrich_policy = "hard_cases_only"
elif value_report.should_enable_record_enrich is False:
    record_enrich_policy = "conflict_only"
else:
    record_enrich_policy = "enabled"
```

然后在流程中体现：

```text
ModelRouter
  ↓
LLMValueGate
  ↓
StagePolicy
  ↓
LLMEnrichStage
```

这样 ablation 结果才能真正反向控制 Agent 行为。

# 七、第六个优化点：把 EntityGraph 放到流程前半段，而不是只在 pipeline 后半段使用

现在 EntityGraph 已经支持：

```text
EntityAsset
EntityObservation
EntityRelation
EntityRiskProfile
SQLite persistence
generate_clues()
neighborhood()
cross_source_entities()
related_clues()
```

这很好。但从流程看，它更多是在 `CorrelateStage` 里被写入和生成 graph clues。

建议把 EntityGraph 提升到流程前半段：

```text
Query
  ↓
PreflightIntent
  ↓
EntityGraph Retrieval
  - 按 query 风险词查相关实体
  - 按 risk_profile 查高危实体
  - 按 recent window 查最近活跃实体
  - 按 neighborhood 扩展相关线索
  ↓
EvidenceGap Decision
```

也就是说，EntityGraph 不只是 fresh processing 的结果，也应该参与：

```text
1. 是否需要 live collection
2. source selection
3. clue pool retrieval
4. evidence chain 补全
5. refinement target selection
```

例如：

```text
如果 query 命中“群控工具”：
  先查 entity_graph 中 tool_name=群控、相关 contact/domain
  如果已有跨源 contact+tool+domain 图谱：
    直接生成 graph clue
  否则再采集
```

目标流程：

```text
Clue Pool Retrieval
  +
EntityGraph Retrieval
  +
Semantic Local Retrieval
  ↓
EvidenceGap
  ↓
Collection / Pipeline
```

这会让“结构化实体库”从被动产物变成主动决策依据。

# 八、第七个优化点：Live Collection 应该做“分层采集 + 中途停止”

你已经有 layered collection：

```text
group_sources_by_collection_layer
prioritize_sources_for_investigation
```

但 live collection 目前仍然是：

```text
选 source
  ↓
可选 query rewrite
  ↓
按 layer collect
  ↓
收集到 max_raw_records 或 source 结束
```

建议在每一层采集后加入 **Evidence Recheck**：

```text
collect layer 1
  ↓
quick clean/classify/extract sample
  ↓
更新 evidence gap
  ↓
如果已经满足：
    stop collection
否则：
    collect layer 2
```

尤其是高召回模式，不应该一次性把 `max_raw_records=20000` 当作目标，而应该是：

```text
最小足够证据原则
```

建议流程：

```text
Layer 1: 高可信授权源 / 内部样本
  ↓
EvidenceGap Recheck
  ↓
Layer 2: 论坛 / 社媒公开源
  ↓
EvidenceGap Recheck
  ↓
Layer 3: 低质量噪声源
```

每层后判断：

```text
是否已有 cross-source support？
是否已有 contact/domain/tool 关键实体？
是否已有 actionable clue？
是否达到用户需要的 top_k？
是否超出误报/成本预算？
```

这会降低采集时延和噪声量。

# 九、第八个优化点：把“流程状态”显式化

现在 workflow 里有很多 dataclass：

```text
_RunPlanningState
_RetrievalState
_SemanticLocalState
_LiveCollectionState
_FreshProcessingState
_RefinementState
```

这些状态已经有价值。但状态之间缺少统一的 `FlowDecision`。

建议每个阶段输出：

```python
class FlowDecision(BaseModel):
    next_action: Literal[
        "continue",
        "stop_with_pool",
        "run_semantic_local",
        "run_live_collection",
        "run_fresh_processing",
        "run_refinement",
        "manual_review",
        "no_data"
    ]
    reason: str
    evidence_gap: EvidenceGap | None = None
    budget_status: dict[str, Any] = {}
    quality_status: dict[str, Any] = {}
```

这样流程不再依赖隐式判断：

```python
if semantic_state.should_collect_live and collect_source_records is not None:
```

而是显式：

```python
if decision.next_action == "run_live_collection":
```

这对调试、日志、测试、报告都更清楚。

# 十、第九个优化点：Result mode 和 trace 语义还可以更准确

当前 CLI summary 里会输出：

```text
mode: llm_driven_investigation
```

但 fast 模式或 mock/dry-run 下，实际可能没有真实 LLM 调用。这个命名容易误导。

建议把 mode 改成更贴近流程：

```text
clue_pool_only
pool_plus_entity_graph
pool_plus_semantic_local
provided_records_pipeline
live_collection_pipeline
hybrid_investigation
```

同时 trace 应区分：

```text
llm_call_traces
  成本、耗时、cache、token 统计

llm_item_traces
  每条 record/clue 的处理解释

flow_decision_traces
  为什么跳过/进入某个阶段

safety_traces
  哪些 source 被阻断，原因是什么
```

现在已经有 `llm_call_traces` 和 `llm_item_traces`，下一步建议加：

```text
flow_decision_traces
```

这样你可以很清楚地解释：

```text
为什么没有 live collection？
为什么用了 semantic local？
为什么某个 clue 被 refine？
为什么某个 candidate 没进入 high quality？
```

# 十一、建议的新流程版本

我建议你下一版把主流程调整成这样：

```text
Request
  ↓
PolicyGuard / SourcePolicy baseline
  ↓
PreflightIntent
  - 规则解析 query
  - 不调用 LLM
  ↓
Retrieval Fanout
  ├── CluePoolRetrieval
  ├── EntityGraphRetrieval
  └── SemanticLocalRetrieval
  ↓
EvidenceGapGate
  ├── 已满足 → Promotion / SelectiveRefine / Report
  └── 不满足
        ↓
      ConditionalPlanning
      - 只有需要采集或复杂研判时才 LLM intent/plan
        ↓
      SourceSelection
        ↓
      LayeredCollection
        ↓
      PerLayerEvidenceRecheck
        ↓
      FreshPipeline
      - Clean
      - Dedup
      - Classify
      - Extract
      - LLMEnrich only if ValueGate allows
      - EntityGraphUpsert
      - Correlate
      - Promotion
      - Score
        ↓
      Merge Candidates
        ↓
      RefineTargetSelector
        ↓
      BatchRefine
        ↓
      ReviewRoute / Report
```

核心变化是：

```text
1. 先检索已有资产，再决定是否规划和采集。
2. semantic local / entity graph 不再依赖 live collection。
3. 用 EvidenceGap 驱动流程，而不是用简单 bool。
4. promotion 在 refine 之前进一步收紧。
5. LLMValueGate 直接控制 record_enrich。
6. live collection 分层采集并中途 recheck。
```

# 结论

有优化空间，而且主要集中在 **流程顺序和决策闭环**。

当前流程已经具备完整能力，但还偏：

```text
先规划，再检索；
semantic local 依赖 live collection；
是否采集主要靠 summary reason；
candidate 合并后再统一 refine；
LLMValueGate 还没有完全主导运行时；
EntityGraph 更多是 pipeline 产物，而不是前置决策资产。
```

下一步最关键的改法是：

```text
把流程改成“资产优先、证据缺口驱动、条件规划、分层采集、选择性 refine”。
```
