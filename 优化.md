 主要差距

  1. 线索质量是当前最弱项。启动会明确说最终偏端到端，问“当天诈骗引流相关线索信息”并看线索质量和证据。现在系统能产线索，
     但人工 gold 上召回很低，证据链可复核率为 0，答辩容易被追问。

  2. 分类还有明显误判。分类能力不是没有，而是还不足以支撑“严格符合高质量自动研判”。典型错误集中在低相关、防御语境、工具/
     账号/诈骗/刷单边界混淆。

  3. 真实多源采集证据不够稳。全量数据偏 Telegram/IM，虽然 rerun 证明可以改善，但样本只有 26 条。公众号类 source 也没有形
     成明确独立证据。

  4. 交付文档有指标滞后。docs/交付文档.md:267 仍写旧的 manual F1，而最新 data/eval_manual_heldout_report.json 已更新到
     primary F1=0.7484、entity F1=0.9484、clue F1=0.08。答辩前必须统一口径。

  优化建议

  优先级最高的是把“当天诈骗引流线索”做成稳定验收脚本：固定公开合规 source，完整展示 raw -> clean -> classify -> entity
  -> clue -> evidence trace -> 人工复核建议，并把耗时、LLM 调用、token 预算一起输出。

  第二，集中修线索生成和评估。不要只按实体聚类出线索，要让 clue 对齐人工期望对象：风险类型、关键实体、来源、证据 trace、
  时间窗口、可复核摘要。目标先把 object clue F1 和 evidence reviewability rate 从 0 拉起来。

  第三，基于 manual heldout 的 typical errors 修分类规则和二级标签边界，尤其是低相关、防御语境、返利/刷单、工具/账号交易
  混淆。分类不需要追求全自动无复核，但要降低误报和 unknown/review 负载。

  第四，扩大均衡采集 rerun，不要只用 26 条证明多源。建议做一版 200-500 条的公开/授权均衡采集包，明确覆盖 IM/群组、社媒/
  论坛、垂直/众包/二手，能的话补一个公众号或公开文章类 source。

  第五，更新交付文档和答辩口径。当前最稳的说法是：系统已完成本地公开/授权端到端闭环，实体抽取强，清洗稳定，分类和线索质
  量仍处于可复核增强阶段；不要声称线上泛化、自动处置或线索召回已经充分达标。
