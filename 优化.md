# 2. 最大核心问题：Orchestrator 仍然是系统中心，服务只是“伪拆分”

`src/agent/investigation_orchestrator.py` 现在仍然有大约 **2000 行**。`run()` 本身已经变短，这是好事；但大量核心逻辑还在同一个类里。

更关键的是，`src/agent/services.py` 里的 service 目前多数不是独立服务，而是这种模式：

```python
class RunStatePreparationService:
    def __init__(self, owner: Any) -> None:
        self.owner = owner
```

然后内部大量调用：

```python
self.owner._normalize_policy_override(...)
self.owner._routing_profile_config(...)
self.owner._planner_runtime_context(...)
self.owner._resolve_budget(...)
self.owner._select_sources(...)
```

这不是彻底拆分，只是把 Orchestrator 的方法调用包了一层。核心依赖方向还是：

```text
services 依赖 Orchestrator 内部私有方法
```

这会导致三个问题：

```text
1. service 无法独立测试。
2. Orchestrator 仍然是事实上的上帝类。
3. 新能力还会继续堆回 Orchestrator，因为私有 helper 都在那里。
```

这部分不能再继续“提几个方法”了，应该改成真正的 workflow 架构。

建议目标结构：

```text
src/workflows/
  investigation_workflow.py
  workflow_context.py
  workflow_result.py

src/application/
  investigation_service.py

src/agent/services/
  intent_planning_service.py
  source_selection_service.py
  retrieval_service.py
  semantic_local_service.py
  live_collection_service.py
  fresh_processing_service.py
  clue_refinement_service.py
  telemetry_service.py
```

核心原则是：

```text
Orchestrator 不再被 service 反向依赖。
Service 只依赖显式注入的组件。
Workflow 只串联服务。
```

现在：

```python
RunStatePreparationService(owner=self)
```

应该改成：

```python
RunStatePreparationService(
    intent_parser=intent_parser,
    planner=planner,
    profile_resolver=profile_resolver,
    budget_factory=budget_factory,
    source_selector=source_selector,
)
```

最终 `InvestigationWorkflow.run()` 应该长这样：

```python
class InvestigationWorkflow:
    def run(self, request: InvestigationRequest) -> InvestigationResult:
        context = self.intent_planning.prepare(request)
        retrieval = self.retrieval.retrieve(context)
        semantic = self.semantic_local.retrieve(context, retrieval)
        live = self.live_collection.collect(context, retrieval, semantic)
        fresh = self.fresh_processing.process(context, retrieval, semantic, live)
        refined = self.clue_refinement.refine(context, retrieval, semantic, live, fresh)
        summary = self.telemetry.build(context, retrieval, semantic, live, fresh, refined)
        return self.result_builder.build(context, summary, refined)
```

验收标准：

```text
InvestigationOrchestrator / InvestigationWorkflow 不超过 300 行。
run() 不超过 60 行。
services.py 不再存在 owner: Any。
每个 service 有单元测试。
所有 service 只依赖显式接口，不调用 Orchestrator 私有方法。
```

这是下一阶段最核心的重构。

# 3. 第二个核心问题：系统仍然有“双主链路”

现在新链路是：

```text
IntelligencePipeline
  → CleanStage
  → DedupStage
  → ClassifyStage
  → ExtractStage
  → LLMEnrichStage
  → CorrelateStage
  → ScoreStage
```

但 `OfflineClueBuilder` 里仍然保留：

```python
if not payload["risk_clues"]:
    result = self.phase_engine.run(...)
    payload = result.model_dump()
```

也就是说，当前系统仍然有两条主链路：

```text
新主链路：IntelligencePipeline
旧主链路：PhaseTwoThreeEngine
```

这不是小问题。它会导致：

```text
1. 一部分样本走新 pipeline，一部分样本 fallback 到旧 engine。
2. evaluation 跑的是 IntelligencePipeline，但线上 investigation 可能 fallback 到 PhaseTwoThreeEngine。
3. 新增规则、LLM enrich、router、budget 可能只对新链路生效。
```

推荐改法：

```text

  ↓
拆成 pipeline stages 或 downstream builders
```

具体拆法：

```text

  → ClassifyStage

PhaseTwoThreeEngine 中的 entity extractor 能力
  → ExtractStage

PhaseTwoThreeEngine 中的 clue aggregator 能力
  → CorrelateStage

PhaseTwoThreeEngine 中的 playbook builder
  → PlaybookStage

PhaseTwoThreeEngine 中的 strategy planner
  → StrategyStage

PhaseTwoThreeEngine 中的 lifecycle manager
  → FeedbackLifecycleService
```

新链路应变成：

```text
Raw
  ↓
CleanStage
  ↓
DedupStage
  ↓
ContextPolarityStage
  ↓
ClassifyStage
  ↓
ExtractStage
  ↓
LLMEnrichStage
  ↓
EntityNormalizeStage
  ↓
CorrelateStage
  ↓
ScoreStage
  ↓
PlaybookStage
  ↓
StrategyStage
  ↓
ReviewRouteStage
```

`OfflineClueBuilder` 最终不应该再 fallback 到旧 engine。它应该只是：

```python
class FreshIntelProcessor:
    def process(self, records, context):
        return self.pipeline.run(records, context=context)
```

如果新 pipeline 没有产出 clue，应该明确返回：

```python
{
  "pipeline_backend": "intelligence_pipeline",
  "fallback_backend": None,
  "no_clue_reason": "aggregation_threshold_not_met"
}
```

而不是悄悄调用旧链路。

验收标准：


这是第二个必须动的核心点。

# 5. 第四个核心问题：Source 安全边界没有进入 Orchestrator 核心

这是当前版本里最严重的实际风险。

我做了一个简单验证：构造一个 source：

```python
{
  "source_name": "bad",
  "source_type": "forum",
  "source_url": "https://example.com/q",
  "query_url_template": "https://example.com/search?q={query}",
  "legal_basis": "UNAUTHORIZED_PRIVATE_GROUP",
  "allow_login_bypass": True,
  "allow_interaction": True
}
```

然后传入一个自定义 `collect_source_records`。结果 Orchestrator 会调用 collector：

```text
COLLECTED UNAUTHORIZED_PRIVATE_GROUP True True
```

也就是说，现在 source 合规边界主要在 `HTTPFeedCollector` 里，但如果调用方注入自定义 `collect_source_records`，就可以绕过 collector 的 source authorization。

这不是小修问题。黑灰产情报 Agent 的 source policy 必须是**编排层硬边界**，不是某个 collector 的内部实现。

建议新增：

```text
src/safety/source_policy_guard.py
```

核心规则：

```python
AUTHORIZED_LEGAL_BASES = {
    "PUBLIC_COMPLIANT_DATA",
    "AUTHORIZED_PARTNER",
    "INTERNAL_AUTHORIZED_SOURCE",
    "THIRD_PARTY_AUTHORIZED_FEED",
}

FORBIDDEN_TRUE_FLAGS = {
    "allow_login_bypass",
    "allow_captcha_bypass",
    "allow_interaction",
    "allow_auto_join",
    "allow_private_group_access",
    "allow_file_download",
}
```

实现：

```python
class SourcePolicyGuard:
    def validate_for_collection(self, source: Mapping[str, Any]) -> None:
        legal_basis = str(source.get("legal_basis") or "").strip()

        if legal_basis not in AUTHORIZED_LEGAL_BASES:
            raise SafetyPolicyViolation("missing_authorized_legal_basis")

        for field in FORBIDDEN_TRUE_FLAGS:
            if source.get(field) is True:
                raise SafetyPolicyViolation(f"{field}_forbidden")

        url = str(source.get("source_url") or source.get("query_url_template") or "")
        if self._contains_credentials(url):
            raise SafetyPolicyViolation("credentials_in_source_url_forbidden")
```

调用点不能放在 collector 里，必须放在：

```text
SourceSelectionService
LiveCollectionService
_collect_records_from_sources
```

建议逻辑：

```python
selected_sources = [
    source for source in selected_sources
    if source_policy_guard.allowed(source)
]
```

如果有 source 被拒绝，`collection_runs` 应记录：

```python
{
  "source_name": "...",
  "status": "blocked_by_source_policy",
  "reason": "missing_authorized_legal_basis"
}
```

验收标准：

```text
任意 collect_source_records 注入都无法绕过 source policy。
未授权 legal_basis 不会进入 collect。
allow_login_bypass=True 不会进入 collect。
allow_interaction=True 不会进入 collect。
collection_runs 有 blocked 记录。
```

这是必须优先做的核心安全改造。

# 6. 第五个核心问题：Profile 没有完整传到数据平面

`fast` / `balanced` / `high_recall` 的预算已经生效，这是控制平面的进步。但数据平面里还有一处策略断层。

`OfflineClueBuilder.set_runtime_controls()` 现在创建：

```python
self.intelligence_pipeline = IntelligencePipeline(
    llm_gateway=llm_gateway,
    budget_controller=budget_controller,
)
```

但没有把当前 `routing_profile` 传进去。

`IntelligencePipeline` 内部默认：

```python
self.model_router = model_router or ModelRouter()
```

而 `ModelRouter()` 默认是：

```python
profile = "balanced"
```

这意味着：

```text
即使外层 routing_profile=fast，
样本级 ModelRouter 仍然可能按 balanced 策略判断。
```

这就是为什么 fast demo 虽然禁用了 intent/plan LLM，但仍然出现了：

```text
llm_classify: 2 calls
clue_refine: budget_exhausted
```

不是说 fast 一定不能做 `llm_classify`，而是现在**数据平面不知道自己正在 fast 模式下运行**。这是策略传播问题。

建议新增一个显式的 `RunPolicyContext`：

```python
class RunPolicyContext(BaseModel):
    routing_profile: Literal["fast", "balanced", "high_recall"]
    budget: RuntimeBudget
    quality_profile: str
    enable_llm_intent_parse: bool
    enable_query_rewrite: bool
    enable_live_collection: bool
    enable_llm_record_enrich: bool
    enable_llm_clue_refine: bool
    min_rule_confidence_for_auto_accept: float
```

然后所有 stage 都读取同一个 policy context：

```python
pipeline.run(records, context=PipelineContext(policy=run_policy))
```

`OfflineClueBuilder.set_runtime_controls()` 应该变成：

```python
self.intelligence_pipeline = IntelligencePipeline(
    llm_gateway=llm_gateway,
    budget_controller=budget_controller,
    model_router=ModelRouter(profile=run_policy.routing_profile),
    policy=run_policy,
)
```

同时 config 中增加：

```yaml
routing_profiles:
  fast:
    enable_llm_record_enrich: false
    enable_llm_clue_refine: true
```

或者更细：

```yaml
fast:
  llm_stage_policy:
    intent_parse: false
    investigation_plan: false
    source_query_rewrite: false
    record_enrich: false
    clue_refine: true
```

验收标准：

```text
fast 模式下 ModelRouter.profile == "fast"。
fast 是否允许 llm_classify 由配置决定，而不是默认 balanced。
execution_summary 输出 llm_stage_policy。
```

这是控制面和数据面打通的核心问题。

# 7. 第六个核心问题：预算账本还没有区分“尝试、允许、实际消耗”

现在 `LLMGateway` 统计和 `BudgetController` 统计口径不一致。

fast demo 中：

```text
llm_gateway.estimated_tokens = 5946
budget_controller.estimated_tokens = 2487
```

原因是：

```text
LLMGateway 统计包含 budget_denied 的 attempted call；
BudgetController 只统计实际 consume 的 allowed call。
```

这两个指标都有价值，但现在字段名不清楚，容易误解。

应该把预算账本升级成 `RunBudgetLedger`，明确区分：

```text
attempted_calls
allowed_calls
denied_calls
successful_calls
failed_calls
cache_hit_calls
estimated_attempted_tokens
estimated_consumed_tokens
estimated_saved_tokens_by_cache
estimated_denied_tokens
```

建议结构：

```python
class BudgetLedger(BaseModel):
    attempted_calls: int = 0
    allowed_calls: int = 0
    denied_calls: int = 0
    cache_hit_calls: int = 0
    network_calls: int = 0
    successful_calls: int = 0
    failed_calls: int = 0

    attempted_tokens: int = 0
    consumed_tokens: int = 0
    denied_tokens: int = 0
    cached_tokens: int = 0

    by_stage: dict[str, StageBudgetStats]
```

最终 summary 应该显示：

```python
"llm_budget": {
  "attempted_calls": 3,
  "allowed_calls": 2,
  "denied_calls": 1,
  "consumed_tokens": 2487,
  "denied_tokens": 1659,
  "cache_hit_calls": 0,
  "by_stage": {
    "llm_classify": {...},
    "clue_refine": {...}
  }
}
```

这样才是真正的成本治理，而不是两个系统各算各的。

其中 hard negative 必须覆盖：

```text
警方反诈提醒
```

# 9. 第八个核心问题：实体库和图谱还不是一等公民

课题产出明确包含“结构化实体库”。但当前系统主要围绕 clue 流转：

```text
ClueRepo
candidate_clues
clue_retriever
clue_refine
```

实体虽然能抽取，也有 `entity_repo.py`、`graph_repo.py`，但它们没有成为 investigation 的核心检索和关联基础。

现在风险线索主要靠：

```text
RiskClueAggregator
  shared_contact_48h
  shared_domain_multi_source
  repeated_template
```

这是 clue 级聚合，还不是实体情报库。

建议把核心数据模型从“clue-first”升级到：

```text
entity-first + clue-as-view
```

也就是先构建实体资产，再从实体资产生成线索。

目标数据流：

```text
ExtractedEntity
  ↓
EntityNormalizer
  ↓
EntityStore.upsert(entity)
  ↓
EntityObservationStore.add(record_id, entity_id, source, time)
  ↓
RelationBuilder.build_relations()
  ↓
RiskGraph
  ↓
ClueGenerator.generate_from_graph()
```

新增核心表 / repo：

```text
EntityRepository
  entity_id
  entity_type
  canonical_value_hash
  masked_display_value
  aliases
  sensitivity_level
  first_seen
  last_seen

EntityObservationRepository
  observation_id
  entity_id
  trace_id
  source_name
  source_type
  publish_time
  evidence_span
  confidence

EntityRelationRepository
  relation_id
  src_entity_id
  dst_entity_id
  relation_type
  evidence_trace_ids
  confidence

RiskGraphRepository
  neighborhood(entity_id)
  connected_components()
  cross_source_entities()
```

然后 clue 生成从图谱来：

```text
同一 contact 跨 source 出现
同一 domain 关联多个账号
同一工具名 + 同一联系方式
同一话术模板 + 不同账号
同一短链跳转域名
```

当前 `RiskClueAggregator` 可以保留，但应该变成：

```text
GraphClueGenerator
```

而不是直接在 records/entities 上一次性聚合。

验收标准：

```text
同一个 contact/domain/tool 能跨多次 run 归一化。
clue 可以追溯到 entity observations。
clue_retriever 不只检索 clue，也能基于 entity graph 检索。
```

这是从 demo 分析系统走向情报系统的关键跃迁。

# 10. 第九个核心问题：配置化还不够彻底

你已经有：

```text
config/routing_profiles.yaml
config/label_schema.json
config/theme_synonyms.yaml
```

但风险规则、黑话、实体模式、上下文极性仍有大量硬编码。比如：

```text
FineGrainedIntentClassifier
AdvancedEntityExtractor
SlangDictionary
RiskClueAggregator
```

下一步不要继续在代码里加规则。应该做一个 `RuleRegistry`。

目标配置：

```text
config/risk_taxonomy.yaml
config/entity_patterns.yaml
config/slang_dictionary.yaml
config/context_polarity.yaml
config/clue_generation_rules.yaml
config/model_stage_policy.yaml
```

例如：

```yaml
context_polarity:
  defensive_markers:
    - 警方
    - 反诈提醒
    - 安全研究
    - 治理案例
    - 不提供
    - 不要参与
    - 曝光
    - 复盘

risk_taxonomy:
  account_trade:
    name: 账号买卖
    severity: high
    primary_terms:
      - 批量出号
      - 成品号
      - 实名号
      - 老号
    required_entity_any_of:
      - contact
      - price
      - platform
```

对应代码：

```python
class RuleRegistry:
    def load_taxonomy(...)
    def load_entity_patterns(...)
    def load_slang_dictionary(...)
    def load_context_polarity(...)
    def version_hash(...)
```

所有分类器和抽取器都从 registry 读规则。

验收标准：

```text
新增一个风险类型不改 Python 代码。
新增一个黑话不改 Python 代码。
新增一个 hard negative marker 不改 Python 代码。
evaluation 输出规则版本号。
```

# 11. 第十个核心问题：包结构仍然不是真正的产品包

现在 `pyproject.toml` 仍然是：

```toml
[tool.setuptools.packages.find]
include = ["src*", "storage*"]
```

也就是说运行 import 还是：

```python
from src.agent import ...
from storage import ...
```

这对于毕业设计 / demo 可以接受，但对一个长期项目不理想。下一阶段如果要“改到核心”，应该完成 package-root 迁移：

```text
src/blackagent/
  agent/
  application/
  backend/
  collector/
  config/
  domain/
  infra/
  interfaces/
  pipeline/
  retrieval/
  safety/
  scheduling/
  storage/
  workflows/
```

然后：

```python
from blackagent.workflows import InvestigationWorkflow
from blackagent.domain import IntelRecord
from blackagent.pipeline import IntelligencePipeline
```

`pyproject.toml` 改成：

```toml
[tool.setuptools.packages.find]
where = ["src"]

```

同时把 CLI 拆成：

```text

src/blackagent/interfaces/cli/main.py
```

`scripts/run_agent_cli.py` 只保留兼容入口：

```python
from blackagent.interfaces.cli.main import main

if __name__ == "__main__":
    raise SystemExit(main())
```

这一步不是表面命名，它会迫使你整理模块边界。

# 13. 最终判断

你这版已经把上一轮“工程治理层”的建议基本做完了。现在继续修小点已经不是主要矛盾。

真正要往核心改，优先级应该是：

```text
1. SourcePolicyGuard 上升到工作流硬边界。
2. RunPolicyContext 贯穿控制面和数据面。
3. IntelligencePipeline 成为唯一主链路，PhaseTwoThreeEngine 退出 fallback 主流程。
4. Domain contracts 替换裸 dict 数据流。
5. EntityStore / EntityGraph 成为核心资产，clue 从实体图谱生成。
6. Evaluation 从 smoke test 升级为质量门禁。
```

一句话总结：

```text
下一阶段不要再围绕 Orchestrator 补功能。
要把系统改成：
Workflow 控制面 + Typed Pipeline 数据面 + Source/LLM 安全边界 + Entity-first 情报库 + Evaluation 质量门禁。
```

这才是从“能跑的 Agent demo”升级为“可控、可评估、可演进的黑灰产情报分析系统”的核心改造。
