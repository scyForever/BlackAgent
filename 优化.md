## 1. 最大问题：二级标签体系还没有真正闭环

当前 evaluation 有一个非常关键的信号：

```text
primary_classification_f1 = 1.0
secondary_classification_f1 = 0.0
hierarchical_classification_f1 = 0.0
```

我看了 gold 数据，`gold_classification.jsonl` 里主要标的是一级风险类型，例如：

```text
工具交易
诈骗引流
```

但 gold 里几乎没有二级标签。与此同时，模型会输出大量 `secondary_label`，导致：

```text
secondary.fp = 110
hierarchical.fp = 100
hierarchical.fn = 100
```

这说明现在不是模型一定错，而是**标签体系的评估粒度没有统一**。

你必须做一个架构级选择：

```text
方案 A：当前阶段只评估一级风险分类。
  secondary_label 只作为辅助说明，不进入质量门禁。

方案 B：正式做层级分类。
  gold 数据必须补齐 expected_secondary_risk / expected_secondary_labels。
  模型输出 secondary_label 才能被正式评估。
```

不要继续维持现在这种状态：

```text
模型输出二级标签
gold 不标二级标签
evaluation 又把二级标签当成错误
```

这会让质量门禁失真。

建议改法：

```text
1. 在 evaluation 里显式区分 primary-only 和 hierarchical 模式。
2. 如果 gold 没有 secondary 标签，则 secondary 指标输出为 not_applicable，而不是 0.0。
3. 如果启用 hierarchical 模式，则 gold 必须补齐二级标签。
4. 模型输出中的 secondary_label 如果置信度不足，应进入 candidate_secondary_labels，而不是 final secondary_label。
```

最终结构建议是：

```json
{
  "classification": {
    "primary": {
      "label": "工具交易",
      "confidence": 0.91
    },
    "secondary": {
      "final": null,
      "candidates": [
        {
          "label": "群控脚本",
          "confidence": 0.62,
          "evidence": ["群控", "脚本"]
        }
      ]
    },
    "granularity": "primary_only"
  }
}
```

这是现在最优先的质量问题。否则你会看到一级分类很好，但层级分类指标一直是 0。

## 3. LLM trace 语义仍然混合：一次 batch 调用会产生多条 clue_refine trace

从 high_recall demo 看：

```text
llm_gateway.call_count = 3
llm_budget.allowed_calls = 3
```

但 `llm_traces` 里会出现多条：

```text
clue_refine
clue_refine
clue_refine
...
```

原因是 `LLMClueRefiner.refine_batch()` 是一次 LLM 调用处理多条 clue，但它为每条 clue 生成一条 trace。这是合理的，但现在 trace 名字和 gateway call 统计混在一起，容易误解为“调用了很多次 LLM”。

建议把 trace 分成两类：

```text
llm_call_traces:
  一次 gateway 请求一条。
  用于成本、token、耗时、cache、错误统计。

llm_item_traces:
  一条 clue / record 一条。
  用于解释每个 item 是否 fallback、是否被 refine、是否 review_required。
```

当前输出可以改成：

```json
{
  "llm_call_traces": [
    {
      "stage": "clue_refine",
      "mode": "batch",
      "call_id": "llm_call_001",
      "item_count": 9,
      "ok": true,
      "cache_hit": false,
      "budget_tokens": 5209
    }
  ],
  "llm_item_traces": [
    {
      "stage": "clue_refine_item",
      "call_id": "llm_call_001",
      "clue_id": "...",
      "used_fallback": true
    }
  ]
}
```

这样效果 / 成本 / 时延统计不会和解释 trace 混在一起。


```python
class PipelineResult(BaseModel):

    candidate_clues: list[CandidateClue]
    actionable_clues: list[ActionableClue]
```

这是下一步很核心的架构收口。

# 最终判断

这版已经很接近一个可交付的系统原型。剩下的优化空间不是“有没有 Agent 能力”，而是：

```text
1. 二级标签体系是否真正可评估。
2. LLM 是否真的产生边际收益。
3. trace / token / budget 统计是否语义清楚。
4. pipeline 是否彻底 typed-first。
5. EntityGraph 是否真正成为结构化情报库。
6. 规则是否真正可运营。
7. 评估集是否足够困难，能证明 LLM + NLP 协同价值。
```
