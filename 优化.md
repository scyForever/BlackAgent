三、这版仍然存在的关键问题
1. 顶层 mode 仍然不准确

CLI 输出里仍然是：

mode: llm_driven_investigation

但 execution_summary 里是：

mode: hybrid_investigation

现在这个顶层 mode 已经过时了。当前系统已经不再是简单 LLM-driven，而是：

hybrid investigation
规则 / NLP / 图谱 / LLM 组合流程

建议把顶层 mode 改成真实流程模式：

hybrid_investigation
provided_records_pipeline
pool_plus_semantic_local
pool_plus_entity_graph
live_collection_pipeline
asset_first_investigation

至少不要继续叫：

llm_driven_investigation

这个命名会误导评审，让人以为系统主要靠 LLM 驱动，而你现在实际已经做成了规则、图谱、LLM 的混合架构。

2. Graph clue 默认关闭，导致图谱评估不直观

如上所说，cross_source_entity_graph.jsonl 如果直接作为 --clues-gold 跑，默认结果是：

expected_clue_count = 1
actual_clue_count = 0
clue_f1 = 0.0

但手动在 pipeline context 中打开：

enable_graph_clue_generation=True

就可以生成正确的：

entity_graph_tool_trade_cluster

这说明问题不是图谱逻辑完全失败，而是流程开关没有进入配置和评估脚本。

建议：

1. 在 RunPolicyContext 增加 enable_graph_clue_generation。
2. 在 routing_profiles.yaml 配置它。
3. evaluate_pipeline.py 在 clue gold 包含 graph clue 时自动打开。
4. execution_summary 输出 graph_clue_generation_enabled。

否则你会在答辩或评审时遇到一个问题：

代码里有 EntityGraph，但评估里图谱线索跑不出来。
3. EntityGraph 的 risk profile 持久化仍有漏洞

EntityGraphStore 已经支持 SQLite 持久化：

entity_asset
entity_observation
entity_relation

但我检查到一个关键问题：_observation_risk 是内存字典：

self._observation_risk: dict[str, str] = {}

add_observation() 会写：

self._observation_risk[observation_id] = risk_category

但 SQLite 的 entity_observation 表里没有 risk_category 字段，也没有单独的 observation risk 表。_load_from_db() 加载时也不会恢复 _observation_risk。

结果是：

同一个 EntityGraphStore 当前 run 内 risk_profile 有 risk_categories；
重启或重新加载 SQLite 后，risk_categories 可能丢失；
graph clue 的 risk_category 可能退化为 unknown。

这对“结构化实体库”很关键。建议立即修：

ALTER TABLE entity_observation ADD COLUMN risk_category TEXT;

或者新增：

entity_observation_risk
  observation_id
  risk_category

然后：

_persist_observation(observation, risk_category=...)
_load_from_db() 时恢复 self._observation_risk

验收测试应覆盖：

1. 创建 EntityGraphStore(db_path)
2. 写入两个带 risk_category 的 observation
3. close
4. 重新打开 EntityGraphStore(db_path)
5. risk_profile(entity_id).risk_categories 仍然存在
6. generate_clues() 输出正确 risk_category

这是这版最值得优先修的“核心 bug”。

4. Context conflict 评估口径仍需要整理

你已经新增了：

context_conflict.jsonl
hard_slang_ambiguous.jsonl
low_evidence_high_risk.jsonl
cross_source_entity_graph.jsonl
llm_required_cases.jsonl

这是好事。

但我直接跑：

python scripts/evaluate_pipeline.py \
  --gold tests/evaluation/context_conflict.jsonl

会得到：

classification_f1 = 0.0

原因是这些样本的 expected_risk_categories 为空，它们本质上是 hard negative / context conflict negative，而不是普通 classification gold。

建议评估脚本明确区分：

classification gold：必须有 expected_risk_categories
hard negative：expected_risk_categories 为空，计 TN / FP
mixed gold：同时支持 positive 和 negative

当前你可以有两种处理方式：

方案 A：
  context_conflict.jsonl 作为 --hard-negative 输入，不作为 --gold 输入。

方案 B：
  evaluate_pipeline.py 支持 mixed mode：
    expected_risk_categories 为空 → 作为 negative 计算
    expected_risk_categories 非空 → 作为 positive 计算

否则困难集评估会出现“看起来 F1=0”的误导。

5. Evaluation 里 LLMValueGate 行为容易让人误解

主评估里：

llm_mode = off
llm_calls_per_1000_records = 0

ablation 里：

high_recall_mock llm_calls_per_1000_records = 145

runtime 里如果没有 latest value report，则 high_recall 默认会变成：

hard_cases_only

这三种状态分别是：

评估主线：不用 LLM
ablation：强制测 LLM 价值
runtime：根据 latest value report 或缺省 hard_cases_only 控制

这在工程上是合理的，但输出报告应该明确说明，否则会让读者困惑：

为什么 evaluation 里 llm_value_gate 是 enabled？
为什么 ablation 里 LLM 调了很多？
为什么 runtime 又说 hard_cases_only？

建议在 evaluate_pipeline.py 输出里增加：

"llm_value_gate_mode": "disabled_for_ablation_measurement"

或者：

"runtime_value_gate_applied": false

这样口径会更清楚。

6. llm_traces 仍混有 model_route 解释 trace

CLI summary 里现在有：

model_route
model_route
...
clue_refine
clue_refine_item

execution_summary 已经分了：

llm_call_traces
llm_item_traces

这是好的。但顶层 llm_traces 里仍混合：

intent_parse
investigation_plan
model_route
clue_refine
clue_refine_item

model_route 不是 LLM 调用，只是路由解释。建议最终拆成：

llm_call_traces
llm_item_traces
model_route_traces
flow_decision_traces
safety_traces

不要再把所有东西都塞进 llm_traces。否则评审看 trace 时会误以为 model_route 也是 LLM 调用。

7. flow_decision_traces 在 provided records 路径里为空

我跑 --demo-sample 和 --routing-profile high_recall 时，因为 demo records 是 provided records，所以走的是 _build_planned_context()，最终：

flow_decision_traces = []

但这个路径其实也应该有 flow decision，例如：

{
  "stage": "provided_records",
  "next_action": "run_fresh_processing",
  "reason": "user_provided_records_bypass_asset_preflight"
}

这不影响功能，但影响可解释性。建议所有路径都输出 flow decision，包括：

provided_records
preflight_satisfied
preflight_insufficient
semantic_local_satisfied
live_collection_disabled
live_collection_started
fresh_processing_started
refine_skipped

这样 Agent 的执行流程更像一个可审计工作流，而不是一串隐含判断。


五、下一步最值得修的 6 个点
1. 修 EntityGraph risk profile 持久化

这是最优先的。

现在 SQLite 持久化没有保存 observation 对应的 risk_category。建议加字段或新表，确保重启后：

risk_profile.risk_categories 不丢失
generate_clues() 不退化为 unknown
2. 把 graph clue generation 纳入 profile 配置

建议在 RunPolicyContext 加：

enable_graph_clue_generation: bool

并在配置里控制：

balanced:
  enable_graph_clue_generation: true

high_recall:
  enable_graph_clue_generation: true

同时 evaluate_pipeline 在 graph clue gold 中自动打开。

3. 修顶层 mode 命名

把：

llm_driven_investigation

改成：

hybrid_investigation

或者更细：

provided_records_pipeline
asset_first_investigation
live_collection_pipeline
4. 整理 evaluation 的 negative / mixed gold 口径

让 context_conflict.jsonl 这种空 expected risk 的数据不再作为普通 classification positive 评估，而是进入：

hard_negative
context_negative
mixed_gold
5. 把 flow_decision_traces 覆盖所有路径

尤其是：

provided_records
preflight_satisfied
preflight_insufficient
semantic_local_satisfied
live_collection_disabled
refine_skipped

这会让 Agent 流程非常容易解释。

6. 收敛 trace 命名

把：

llm_traces

拆成：

llm_call_traces
llm_item_traces
model_route_traces
flow_decision_traces
safety_traces

这会避免“路由解释 trace”和“真实 LLM 调用 trace”混在一起。

最终评价

这版代码已经明显超过“能跑 demo”的阶段。它已经有比较完整的 Agent 工作流、评估体系、source 安全边界、LLM 成本控制、typed pipeline 和实体图谱基础设施。

但如果你问“还有没有优化空间”，我的判断是：

有，但主要集中在交付收口和生产化细节，而不是主链路能力缺失。

最值得优先修的是：

1. EntityGraph risk_category 持久化；
2. graph clue generation 默认策略；
3. 顶层 mode / trace 语义；
4. evaluation negative/mixed 口径；
5. flow_decision_traces 全路径覆盖。

这几个修完后，这版就非常适合作为“黑灰产情报分析 Agent 的阶段性交付版本”。
