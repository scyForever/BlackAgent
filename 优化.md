 逐项判断

  - 数据采集：基本符合，但不够均衡。当前有 4163 条 raw、83 个 source、覆盖 IM/群组、社媒/论坛、垂直/技术三类；但全量数据
    明显被 IM/Telegram 类主导，vertical_or_technical 只有 21 条。答辩可以说“打通 3 类”，但不能说“多源天然均衡”。

  - 智能清洗：基本符合。当前清洗后 3464 条，丢弃 699 条，其中重复丢弃 650 条，高危语料 1095 条。去重、噪声过滤、高危识别
    都已落地。

  - 意图分类：功能符合，效果不严格达标。全量 3464 条都有分类，但仍有 unknown=190、待研判=398、复核 970 条。更关键的是人
    工 held-out 上 primary F1=0.6259、secondary F1=0.5088、hierarchical F1=0.4216、FPR=0.4454，离“高质量线索”要求还有明
    显差距。

  - 实体抽取：整体较强，但有伪实体问题。人工 held-out entity F1=0.9875，说明链接、联系方式、账号、工具等抽取能力较稳；但
    当前 top entities 里仍有 Channel、Image、https、社交 footer URL、站点模板链接等伪实体，说明后处理还没完全压住页面模
    板噪声。

  - 风险线索/证据链：有实现，但验收证据不足。代码有 clue promotion、entity graph、evidence reviewability；但 data/
    eval_manual_heldout_report.json 中 clue gold 是 not_applicable_no_gold，当前不能强声称“人工 held-out 线索质量已验
    证”。

  - Agent / 成本 / 时延：方向符合。规则快轨、预算控制、LLM value gate、query rewrite、历史线索池、live collection 分支都
    存在；并且默认 LLM/网络/处置都受控。问题是当前离线评估里 LLM 基本关闭，
    llm_added_cost_without_measured_quality_gain，还没有证明真实 LLM 在困难样本上带来稳定增益。

  - 合规与安全：符合度较高。采集必须显式 network.enabled=true，source 需要 legal basis，禁止 bypass/proxy/captcha 等语
    义；处置默认 disabled/dry-run，敏感实体有 mask/hash，Agent 不能自动写正式库或自动下发策略。

  最需要优化的地方

  1. 优先提升分类质量。围绕 data/eval_manual_heldout_report.json 的 typical errors 加 hard negative：普通教程/技术讨论、
     X/Tieba 公共资料、Steam/mod/automation 语义歧义、返佣/刷单/引流混淆、账号交易与工具交易交叉。当前这是最大扣分点。

  2. 把实体后处理接到交付出口。重点过滤页面导航、社交 footer、模板 URL、Image/Channel/https 这类伪实体；高价值实体优先保
     留 TG/微信/QQ、域名、账号、邀请码、结算方式、工具名。

  3. 强化 source 配额采集。给 vertical_or_technical、贴吧/论坛、二手交易、众包平台设置最低配额和单 source 上限，避免全量
     语料被单一 Telegram/Automationforum 类来源主导。

  4. 补线索 gold 评估。把 tests/evaluation/manual_heldout_clues.jsonl 真正纳入当前报告，至少做 20-50 条人工线索 gold，评
     估 clue precision/recall、重复率、证据链可复核性。

  5. 做黑话候选闭环。当前有 slang candidate report，但还应形成“候选 → 人工确认 CSV → gray rollout → activate → 评估增
     益”的复跑脚本和报告。

  6. 准备一个真实端到端答辩脚本。用类似“取当天诈骗引流相关线索”作为 query，显式展示 source 选择、采集、清洗、分类、实
     体、线索、证据链、成本/时延摘要。
